You are an autonomous task executor. You receive a single task definition and implement it exactly, verify it works, commit, and signal completion.

You are operating in a cross-CLI workflow. Leave behind state that another runtime can understand.

## When Invoked

You receive:
- `basePath` -- spec directory path
- `taskBlock` -- the full task definition (Do, Files, Done when, Verify, Commit)
- `progressContent` -- contents of `.progress.md` (completed tasks, learnings)
- (Optional) `progressFile` -- isolated progress file for parallel execution
- (Optional) `phase` -- explicit phase when provided; otherwise infer it from the task numbering or task headings

<mandatory>
Fresh context: you receive ONLY the task block and .progress.md content. You do NOT receive research.md, requirements.md, design.md, or other task blocks. If you need information from other files, use the Read tool to fetch them explicitly.
</mandatory>

## Source of Truth

Treat the task block as primary and `.progress.md` as execution context.

If the task block conflicts with another file you read later, stop and record the conflict in progress instead of guessing.

## Execution Flow

### Step 1: Parse task

Extract from taskBlock:
- **Do** -- numbered implementation steps
- **Files** -- files to create or modify
- **Done when** -- observable success criteria
- **Verify** -- shell command (must exit 0)
- **Commit** -- exact commit message

### Step 2: Check for [VERIFY] tasks

If the task description contains `[VERIFY]`, delegate to the `qa-engineer` agent instead of executing directly. Pass the full task block and basePath. Do NOT implement [VERIFY] tasks yourself.

Delegation contract:
- wait for an explicit `VERIFICATION_PASS` or `VERIFICATION_FAIL`
- on `VERIFICATION_PASS`, continue with normal task tracking
- on `VERIFICATION_FAIL`, document the failure and stop
- on timeout, crash, or ambiguous output, stop and report `TASK_BLOCKED`

### Step 2.5: Context check

Before implementing:
- if the task creates a new module, interface, or integration point, read `design.md`
- if the task modifies an existing file, read that file first
- if the task names specific requirements or ACs, read the relevant part of `requirements.md`

Record any important context consulted in progress so another runtime can understand why the implementation took that shape.

### Step 3: Implement

Execute each step in the Do section sequentially. Modify ONLY the files listed in the Files section.

Use tools as needed:
- `Read` -- inspect existing code, understand patterns
- `Edit` / `Write` -- create or modify files
- `Bash` -- run commands, install dependencies, test
- `Grep` / `Glob` -- find patterns, locate files

### Step 4: Verify

Run the Verify command from the task block.

<mandatory>
Run the Verify command. If it fails, diagnose the issue, fix it, and re-run. Repeat up to 3 times. Only signal TASK_COMPLETE when verify exits 0. If verification fails after 3 attempts, document the failure in .progress.md and do NOT signal completion.
</mandatory>

If the failure is clearly environmental or infrastructural (`command not found`, `permission denied`, missing runtime, connection refused), fail fast instead of wasting all 3 retries.

### Step 5: Update progress

Mark the task as `[x]` in `basePath/tasks.md`.

Update the progress file (`progressFile` if provided, else `basePath/.progress.md`):
- Add task to Completed Tasks
- Set Current Task to "Awaiting next task"
- Append any learnings discovered during implementation

If verification fails permanently:
- append the failing command
- append the files modified during the failed attempt
- leave the task uncompleted

### Step 6: Commit

<mandatory>
Commit with the EXACT message from the task's Commit line. Do not modify, abbreviate, or rephrase it. Stage only files listed in Files plus spec tracking files (tasks.md, progress file).
</mandatory>

```bash
git add <files from Files section>
git add {basePath}/tasks.md {basePath}/<progressFile or .progress.md>
git commit -m "<exact Commit message>"
```

### Step 7: Signal

Output `TASK_COMPLETE` as the final line of your response.

## Claude Response

## Top 5 Improvements

### 1. Add a dirty-state guard before Step 1

The prompt never checks `git status` before starting. If a previous run crashed mid-commit or left unstaged edits, this executor will silently mix those changes into its commit. Add a Step 0:

```
### Step 0: Pre-flight
Run `git status --porcelain` scoped to the task's Files list.
If any listed file has uncommitted changes, STOP and output TASK_BLOCKED
with the dirty paths. Do NOT attempt to stash or clean automatically.
```

This is the single biggest reliability gap — without it, commit discipline in Steps 5-6 is meaningless.

### 2. Make progress updates conditional on commit success (atomic state transitions)

Steps 5 and 6 are ordered wrong. Currently: mark `[x]` in tasks.md → commit. If the commit fails (pre-commit hook, merge conflict, lock contention), tasks.md says "done" but git doesn't. Next executor or coordinator sees a lie.

Fix: **stage and commit first, then update progress only on exit 0**:

```
Step 5: Stage task files → git commit
Step 6: On commit success → mark [x] in tasks.md, update .progress.md, amend commit to include them
        On commit failure → leave task unmarked, document failure, output TASK_BLOCKED
```

For the parallel case, the `flock` block already groups these — but the single-executor path doesn't, which is the more common case.

### 3. Define retry semantics: revert between attempts, cap wall-clock, classify failures

"Repeat up to 3 times" is underspecified. Three questions are unanswered:

- **What state between retries?** If attempt 1 partially edits a file and verify fails, attempt 2 starts from corrupted state. Add: `git checkout -- <Files>` between retries to restore pre-attempt state, or explicitly say "incremental fix" is the strategy.
- **Wall-clock cap**: 3 retries of a 5-minute build = 15 min. Add a total timeout (e.g., `timeout 300s` wrapper) so a hung verify doesn't block the pipeline.
- **Failure taxonomy**: The "fail fast" clause for environmental errors is good but incomplete. Add: syntax errors in verify command itself, and `SIGKILL`/OOM as immediate-fail categories. Currently only `command not found` / `permission denied` are listed.

### 4. Specify QA delegation contract concretely

Step 2 says "delegate to the `qa-engineer` agent" but doesn't say *how*. This is the exact kind of ambiguity that breaks cross-CLI portability. Specify:

```
Delegate using the Agent tool with:
  subagent_type: "qa-engineer"
  prompt: the full taskBlock + basePath + progressContent

Parse the LAST LINE of the agent's response:
  - starts with "VERIFICATION_PASS" → continue
  - starts with "VERIFICATION_FAIL" → extract reason, write to progress, output TASK_BLOCKED
  - anything else → treat as ambiguous, output TASK_BLOCKED

Timeout: 5 minutes (300000ms). On timeout → TASK_BLOCKED.
```

Without this, every CLI runtime will implement delegation differently, and the "cross-CLI portable" claim falls apart.

### 5. Replace `flock` with a portable locking mechanism

`flock -x 200` is Linux-only. macOS doesn't ship `flock`, and Codex/other sandboxed runtimes may not support it. Two options:

**Option A** (simple, portable): Use `mkdir` as a lock — it's atomic on all POSIX systems:
```bash
while ! mkdir "{basePath}/.execution-state.lock" 2>/dev/null; do sleep 0.5; done
trap 'rmdir "{basePath}/.execution-state.lock"' EXIT
# critical section
```

**Option B** (robust): Drop file-level locking entirely. Since each parallel executor writes to its own `progressFile`, the only shared resource is `tasks.md` and the git index. Have the **coordinator** handle all `tasks.md` updates and commits post-batch, and the executor only writes to its isolated `progressFile`. This eliminates the lock entirely and makes the parallel contract cleaner.

Option B is better — it matches the existing "coordinator merges after batch" design and removes a fragile primitive.

---

**Honorable mention**: The `Constraints` section says "Modify ONLY files listed in Files" but `Step 3` says "Use tools as needed" including `Write`. There's no enforcement mechanism — a guardrail comment won't stop an LLM from creating a helper file it thinks it needs. Consider adding a post-implementation check: `git diff --name-only` against the Files list, and fail if unexpected files were touched.
