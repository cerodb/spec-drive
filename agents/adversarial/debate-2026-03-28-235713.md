## Round 1 — CRITIC

# Critique: `researcher` Agent Specification

**Content type**: Agent behavioral specification (prompt-as-spec). "Improved" means: the agent reliably produces correct, complete, actionable research under real-world conditions — including when things go wrong.

---

## 1. The `jq` State Update Is a Data-Loss Bomb

**Section**: "Final Step: Set Awaiting Approval"

```bash
jq '.awaitingApproval = true' {basePath}/.spec-drive-state.json > /tmp/sd-state.json && mv /tmp/sd-state.json {basePath}/.spec-drive-state.json
```

**Failure scenario**: If `.spec-drive-state.json` doesn't exist yet (first run, deleted by user, or another agent failed to initialize it), `jq` reads nothing and outputs `{"awaitingApproval": true}` — nuking every other field that should be in the state file. If two agents run concurrently and both write to `/tmp/sd-state.json`, one overwrites the other's temp file. The hardcoded `/tmp/` path is a classic race condition.

**Fix**: Use `mktemp` for the temp file. Add a guard that checks the state file exists and contains valid JSON before mutating it. Define what the expected schema of the state file is, and validate after write.

---

## 2. No Error Handling Model Whatsoever

The spec says "If idea.md is missing or empty, stop immediately and report the error." That's the **only** error path defined. Every other step assumes success.

**Failure scenarios**:
- **WebSearch returns nothing or errors** (rate limit, network down, API key invalid). The agent has no instruction on what to do. Does it skip? Halt? Fabricate?
- **Codebase is empty or has no matching files**. Glob returns zero results. The agent could silently produce an empty "Codebase Analysis" section, and downstream agents would assume the codebase was inspected and found clean.
- **`{basePath}/.progress.md` doesn't exist** when the agent tries to append. The "append only" instruction fails.
- **`{basePath}/.spec-drive-state.json` is malformed JSON**. The `jq` command fails silently (the `&&` prevents the `mv`, but the agent thinks it succeeded).

**Fix**: Define explicit behavior for each step's failure mode. At minimum: "If WebSearch returns no results, record 'No external research found — manual review required' and flag it as an open question." Same pattern for every step.

---

## 3. The Quality Commands Section Has a False Sense of Completeness

**Section**: Step 3, Quality Command Discovery

The spec checks `package.json`, `Makefile`, and `.github/workflows/*.yml`. That's it.

**Failure scenario**: The project uses `pyproject.toml` (Python), `Cargo.toml` (Rust), `go.mk`, `docker-compose.yml` for test services, `turbo.json` or `nx.json` for monorepo task runners, `justfile`, `taskfile.yml`, or `deno.json`. The researcher reports "Not found" for all quality commands, and the task-planner skips all verification — on a project that has full CI coverage via a tool the spec never told the agent to look for.

**Fix**: Either make the source list extensible ("Check these sources, plus any other build/task configuration files found via `Glob: **/Makefile **/Taskfile* **/justfile **/pyproject.toml **/Cargo.toml`") or explicitly state this agent is scoped to Node.js/TypeScript projects only and reject others.

---

## 4. Web Research Is Underspecified to the Point of Uselessness

**Section**: Step 1

The instruction is literally:
```
WebSearch: "[topic] best practices"
WebSearch: "[topic] common pitfalls"
```

**Failure scenario**: `idea.md` says "Build a real-time collaborative editor." The agent searches "real-time collaborative editor best practices" and gets 50 results ranging from CRDTs to OT to Firebase to Yjs. It has zero guidance on how many sources to consult, how to evaluate conflicting advice, how to prioritize relevance, or when to stop searching. One run produces 2 bullet points; the next produces 40. The output is non-deterministic and the downstream product-manager can't rely on coverage.

**Fix**: Define a minimum search breadth (e.g., "at least 3 distinct searches"), a maximum output cap (e.g., "top 5 findings"), and evaluation criteria (e.g., "prioritize official docs > well-known blogs > random results; discard results older than 2 years unless foundational").

---

## 5. Feasibility Assessment Has No Calibration Anchor

**Section**: Step 4

Effort scale: "S = hours, M = 1-2 days, L = 3-5 days, XL = 1+ week."

**Failure scenario**: Hours for whom? A senior engineer? A junior? The AI agent itself? An "S" for a senior Rust developer is an "XL" for someone learning the language. Risk is similarly unanchored — "High" relative to what baseline? The downstream architect receives "Component X: M effort, Medium risk" and has no way to validate or adjust because there's no frame of reference.

**Fix**: Define the persona these estimates target (e.g., "a mid-level developer familiar with the project's stack"). Define risk relative to something concrete (e.g., "High = likely to require design changes or has external dependencies; Med = known approach but non-trivial; Low = straightforward implementation").

---

## 6. "Append Only" to `.progress.md` Creates Unbounded Growth

**Section**: Progress Update

"Append only. Never delete existing learnings."

**Failure scenario**: After 20 research iterations (pivots, re-runs, corrections), `.progress.md` is a sprawling document full of contradictory learnings from different points in time. "Found that library X is ideal" followed 5 entries later by "Library X has a critical CVE, do not use." Downstream agents read all learnings with equal weight and make decisions based on stale information.

**Fix**: Either timestamp each learning entry so consumers can resolve conflicts chronologically, or allow a "supersede" mechanism where new learnings can explicitly mark old ones as obsolete without deleting them.

---

## 7. No Scope Boundary on Codebase Exploration

**Section**: Step 2

```
Glob: **/*.ts, **/*.js, **/*.json — find relevant files
```

**Failure scenario**: The target codebase is a monorepo with 50,000 files. The glob returns thousands of matches. The agent has no guidance on how to scope the search (search from project root? from `basePath`? from a `src/` directory?), how many files to inspect, or when to stop. It either times out, reads 200 files burning tokens, or arbitrarily stops at 5 files and misses the critical pattern in file 6.

**Fix**: Define the search root explicitly (e.g., "the nearest `package.json` parent, or `basePath/../`"). Set a depth/count limit. Instruct the agent to start with high-signal files (README, index files, config) before broad exploration.

## 8. The "Source of Truth" Claim Contradicts the Execution Steps

**Section**: "Source of Truth" vs. Steps 1-3

The spec states: "Treat `idea.md` plus the files you explicitly inspect during research as the only source of truth." But Step 1 instructs the agent to search the web and treat external findings as evidence. Web results are not inspected files — they're third-party claims. This is a philosophical contradiction that creates ambiguity: when web research conflicts with codebase evidence, which wins?

**Fix**: Clarify the hierarchy: "Codebase evidence > official documentation > web research > inference. When sources conflict, flag the conflict as an open question."

---

## 9. No Idempotency Guarantee

The spec never addresses what happens if the researcher agent is run twice on the same spec.

**Failure scenario**: Agent runs, writes `research.md`, sets `awaitingApproval = true`. Human hasn't reviewed yet. Agent is accidentally re-triggered (cron, retry logic, user double-click). It overwrites `research.md` with potentially different results (web search results change daily). The human approved version 1 in their head, but version 2 is now on disk.

**Fix**: Either check if `research.md` already exists and `awaitingApproval` is true (and refuse to overwrite without explicit force), or version the output (e.g., `research-v2.md`) and let the human choose.

---

## 10. The Output Template Doesn't Match the Execution Steps

**Section**: Output template vs. Steps 1-5

Step 5 is "Identify Open Questions." The output template has an "Open Questions" section. But there's no section in the template for Step 4's feasibility assessment narrative — only the table. More critically, there's no explicit mapping between each Step and which template section it populates. The agent has to infer the correspondence.

Additionally, the template's `<!-- comments -->` serve as placeholders but provide weaker guidance than the step descriptions. The "Executive Summary" section appears in the template but has no corresponding execution step — the agent must synthesize it from all other steps, but is never told to do so.

**Fix**: Add a "Step 6: Write Executive Summary — synthesize key findings from Steps 1-5 into 3-5 bullets" step. Map each step explicitly to its output section.

## Round 1 — DEFENDER

# Defense: `researcher` Agent Specification

## The Concern the Critic Missed

Before addressing the ten points: the critic focused almost entirely on **failure modes and edge cases** while missing the most impactful gap in this spec — **there is no triage step that calibrates research depth to project complexity.**

A trivial idea.md ("add a dark mode toggle") gets the same five-step treatment as an ambitious one ("build a distributed event sourcing system"). The agent has no instruction to assess scope FIRST and then decide how many web searches to run, how deeply to explore the codebase, or how granular the feasibility table should be. This matters more than any individual failure mode because it affects **every single invocation** — not just the edge cases. Trivial specs waste tokens on unnecessary research breadth; complex specs get the same shallow treatment. A Step 0 ("Read idea.md, classify complexity as Small/Medium/Large, and scale subsequent research accordingly") would improve output quality more than all ten of the critic's fixes combined.

Now, to the points.

---

## Point 1: The `jq` State Update — Partially Concede

The critic is right that the hardcoded `/tmp/sd-state.json` path is a race condition between concurrent spec executions. I concede this specific defect.

However, the critic's framing as a "data-loss bomb" overstates severity. The `.spec-drive-state.json` file is a lightweight coordination signal, not a data store. If it gets corrupted, you re-run the researcher — you don't lose research.md, which is the actual artifact. The blast radius is "agent coordination breaks," not "data loss."

The "file doesn't exist" scenario is also real but **would only occur on a broken orchestrator**. The spec-drive system initializes this file when creating the spec directory. The researcher doesn't run in isolation — it's invoked by an orchestrator that manages state. Still, defense-in-depth says guard against it.

**Accepted fix, improved:**
```bash
STATE_FILE="{basePath}/.spec-drive-state.json"
if [ ! -f "$STATE_FILE" ] || ! jq empty "$STATE_FILE" 2>/dev/null; then
  echo '{}' > "$STATE_FILE"
fi
TMP=$(mktemp)
jq '.awaitingApproval = true' "$STATE_FILE" > "$TMP" && mv "$TMP" "$STATE_FILE"
```
This uses `mktemp` (fixing the race), validates existing JSON, and initializes gracefully. Better than the critic's suggestion of "define the expected schema and validate after write" — that's over-engineering for a coordination flag.

---

## Point 2: No Error Handling — Mostly Reject

The critic lists four failure scenarios and frames this as "zero error handling." This misunderstands what this document is.

**This is a prompt for an LLM agent, not a program specification.** LLM agents handle ambiguity natively. When WebSearch returns nothing, a competent model doesn't fabricate — and the spec already mandates in its constraints: *"NEVER fabricate sources. If you cannot find information, say 'not found' explicitly."* That IS the error handling for web research failures.

The critic's specific scenarios:

- **WebSearch returns nothing**: Covered by the existing "NEVER fabricate" constraint. The agent records "not found." This is correct behavior.
- **Codebase is empty**: A legitimately empty codebase analysis section IS the correct output — it means "nothing relevant exists yet." The downstream architect needs to know this. An empty section ≠ silent failure; it = accurate finding.
- **`.progress.md` doesn't exist**: Fair point. The agent should create it if missing. I concede this narrow case.
- **Malformed state JSON**: Already addressed in Point 1's fix.

Adding explicit "if X fails, do Y" for every tool invocation would bloat this spec into unreadability and actually **harm** agent performance — over-constrained prompts produce rigid, brittle behavior. The current design trusts the model's intelligence for non-critical paths while mandating behavior for critical ones. That's the right tradeoff.

**Concession**: Add one line: "If `.progress.md` does not exist, create it with an initial `## Learnings` header before appending."

---

## Point 3: Quality Commands Scoped to JS — Reject

The critic claims the spec only checks JS tooling. **The critic missed the project's own constraints.**

From the governing principles (Principle 8): *"Un stack, bien hecho — Node.js runtime, SQLite DB, Next.js+Tailwind UI... Tecnología nueva requiere justificación."* This agent operates within an ecosystem that explicitly mandates a Node.js stack. Checking `package.json` is not a limitation — it's correct scoping.

Moreover, the critic understates what the spec **already covers**:
- `Makefile` — language-agnostic
- `.github/workflows/*.yml` — language-agnostic CI; a Python project's CI file reveals `pytest`, a Rust project's reveals `cargo test`

The CI workflow check alone catches quality commands for ANY language. The critic's hypothetical Rust/Python project would have its commands discovered through the CI step — which the spec already mandates.

Adding a glob for `pyproject.toml`, `Cargo.toml`, `justfile`, etc. sounds reasonable in isolation but violates Principle 1 (*"Simple is smart — no agregar complejidad sin problema real"*) and Principle 9 (*"El código más barato es el que no escribís"*). There is no real problem here given the stack constraint.

**No fix needed.**

---

## Point 4: Web Research Underspecified — Partially Concede

The critic is right that the search examples are thin. But wrong about the diagnosis.

The `[topic] best practices` / `[topic] common pitfalls` lines are **illustrative templates**, not an exhaustive playbook. Over-specifying search strategy makes the agent brittle — a good LLM will naturally decompose "real-time collaborative editor" into searches for CRDTs, OT, Yjs, and operational concerns. Mandating "exactly 3 searches, top 5 findings" produces checkbox compliance, not good research.

That said, the critic correctly identifies that **output bounds** are missing. Without them, one run produces 2 bullets and another produces 40 — that's a real problem for downstream consumers.

**Concession — output bounds only:**
Add to Step 1: "Aim for 3-8 key findings. For each finding, record: what it is (1 line), why it matters for this project (1 line), adopt/adapt/ignore recommendation, source URL. Prioritize official documentation and established references over blog posts."

This constrains the output without constraining the search strategy. The agent remains free to search intelligently while producing predictable output.

---

## Point 5: Feasibility Estimates Unanchored — Reject

The critic asks "hours for whom?" as if these estimates are contractual commitments. They're not. They're **relative sizing signals** for the architect agent.

T-shirt sizing (S/M/L/XL) is a universally understood estimation framework precisely BECAUSE it avoids false precision. Anchoring to "a mid-level developer familiar with the stack" adds a veneer of specificity that doesn't actually improve signal — it just makes the estimate feel more authoritative while being equally uncertain.

What matters is the **relative ordering**: if Component A is "S" and Component B is "XL", the architect knows where to focus design effort. The absolute meaning of "S" matters far less than the delta between components.

The risk scale similarly serves as a triage signal, not a risk register. "High risk" tells the architect "spend more design time here" — which is all it needs to do at the research phase.

**No fix needed.** Adding calibration anchors would create false precision at a stage where precision is neither possible nor useful.

---

## Point 6: Append-Only Creates Unbounded Growth — Reject

The critic posits "20 research iterations" creating contradictory learnings. **This doesn't happen in practice.**

The researcher agent runs **once per spec lifecycle** — occasionally twice if the human requests a re-run after providing feedback. The `.progress.md` file is scoped to a single spec (it lives in `{basePath}/`), and specs have bounded lifetimes. After implementation, the spec directory is archived or deleted.

Twenty iterations of research on the same spec would require 20 human approvals (because the agent sets `awaitingApproval = true` every time). That's not an organic failure mode — it's someone deliberately abusing the system.

The append-only rule exists because **data loss is worse than data noise**. In a multi-agent pipeline, if agent B deletes agent A's learnings, you get silent context destruction. Append-only prevents this class of bug entirely. The tradeoff (theoretical future noise vs. guaranteed present safety) is correct.

**No fix needed.** The critic is optimizing for a scenario that requires 20x the normal iteration count within a single spec lifecycle.

---

## Point 7: Unbounded Codebase Exploration — Partially Concede

The critic is right that `Glob: **/*.ts, **/*.js, **/*.json` on a 50,000-file monorepo is problematic.

However, the critic overlooks that this is a prompt for an LLM, not a script. The agent doesn't execute `cat` on every glob match — it uses the glob results to identify candidate files, then selectively reads the ones relevant to the idea.md vision. LLM agents inherently triage.

That said, explicit guidance on **where to start** would help, especially for less capable models.

**Concession — add scoping guidance to Step 2:**
"Start exploration from the project root (nearest `package.json` ancestor or `basePath/../`). Begin with high-signal files: README, main entry points, configuration files. Then search for files matching key terms from idea.md. Inspect up to 15-20 files maximum — prioritize breadth of pattern discovery over exhaustive coverage."

This bounds the exploration without making it rigid.

## Point 8: Source of Truth Contradiction — Reject

The critic claims a "philosophical contradiction" between the Source of Truth section and Step 1's web research. **This is a misreading.**

The Source of Truth clause says:

> "Treat `idea.md` plus the files you explicitly inspect during research as the only source of truth. Do not rely on: prior chat context, unstated operator intent, hidden tool state."

The purpose is clear: **don't bring in invisible context from prior conversations.** This prevents the agent from saying "as we discussed earlier" or relying on state from a previous session. Web research findings obtained during THIS execution are part of "files you explicitly inspect during research" — the web pages ARE explicitly inspected.

The clause prevents the agent from relying on:
- What the human said in Telegram before invoking the spec
- What another agent wrote in a different spec's directory
- Internal model state from fine-tuning

It does NOT prevent the agent from using its own research findings. There is no contradiction.

When the critic suggests adding a priority hierarchy ("codebase > official docs > web research > inference"), that's actually a useful addition — not because there's a contradiction to resolve, but because conflict resolution guidance helps produce better output. I'd accept it as an enhancement, not a fix.

**Optional enhancement** (not a defect fix): Add to Step 1: "When external research conflicts with codebase evidence, note the conflict explicitly and prioritize codebase reality."

## Point 9: No Idempotency — Partially Concede

The critic's scenario (agent re-triggered while awaiting approval, overwrites research.md) is plausible. I concede the defect exists.

However, the **primary defense** against this is the orchestrator, not this spec. The orchestrator reads `.spec-drive-state.json` before invoking agents — if `awaitingApproval` is true, it shouldn't invoke the researcher. This spec is one layer in a multi-layer system.

That said, defense-in-depth is sound practice. A self-guard is cheap insurance.

**Accepted fix:**
Add to the beginning of the Execution section: "Before starting research, check if `{basePath}/research.md` exists AND `{basePath}/.spec-drive-state.json` has `awaitingApproval: true`. If both conditions are met, stop and report: 'Research already complete and awaiting approval. Use --force to re-run.' This prevents accidental overwrites of approved-but-not-yet-consumed research."

This is simpler and better than the critic's versioning suggestion (`research-v2.md`), which would break downstream agents that expect a fixed filename.

**Claim: "No section in the template for Step 4's feasibility assessment."** Incorrect. The template explicitly contains:

## Round 2 — CRITIC

# Round 2 — CRITIC

The defender conceded five points and proposed fixes for each. Three of those fixes are broken. The defender also introduced a new idea (Step 0 complexity triage) that has its own problems. And three rejections relied on arguments that don't hold up under scrutiny.

---

## A. Weak Fixes That Won't Work

### A1. The Idempotency Guard Is Trivially Bypassed and Wrongly Scoped

**Defender's fix**: "Check if `research.md` exists AND `.spec-drive-state.json` has `awaitingApproval: true`. If both, stop."

**What breaks**: The guard only fires when BOTH conditions are true. But `awaitingApproval` gets set as the LAST action — meaning there's a window between "research.md written" and "awaitingApproval set" where a concurrent re-trigger sees `research.md` exists but `awaitingApproval` is false, and proceeds to overwrite. This is the exact same race condition the defender conceded in Point 1, now reintroduced in the fix for Point 9.

Worse: the `--force` flag the defender proposes doesn't exist anywhere in the spec-drive system. The defender just invented an interface contract without specifying who parses it, how it's passed, or what the orchestrator does with it. It's a fix that delegates to machinery that doesn't exist.

**Actual fix**: Check `research.md` existence ALONE as the primary guard. If it exists and is non-empty, refuse to overwrite. The approval state is a separate concern — don't couple them. And drop the `--force` flag reference until there's an actual mechanism for it.

---

### A2. The `jq` Fix Silently Initializes State, Masking Orchestrator Bugs

**Defender's fix**:
```bash
if [ ! -f "$STATE_FILE" ] || ! jq empty "$STATE_FILE" 2>/dev/null; then
  echo '{}' > "$STATE_FILE"
fi
```

The defender correctly called my "data-loss bomb" framing overblown, then proposed a fix that **actually creates the data-loss scenario I warned about**. If `.spec-drive-state.json` exists but is malformed (maybe another agent was mid-write), this fix nukes it to `{}` — destroying whatever partial state was there. The defender's own argument was "the orchestrator manages this file" — so why is the researcher silently reinitializing it?

If the state file is missing or corrupt, that's an orchestrator bug. The researcher should **fail loudly**, not paper over it. The correct behavior is: validate the file exists and is valid JSON; if not, stop and report "State file missing or corrupt — orchestrator may have failed to initialize."

Silently initializing state violates the defender's own design premise that the orchestrator owns this file.

---

### A3. The Output Bounds for Web Research Create a Perverse Incentive

**Defender's fix**: "Aim for 3-8 key findings."

This sounds reasonable. In practice, it teaches the agent to **pad thin research to meet the minimum** and **truncate rich research to meet the maximum**. When idea.md describes something obscure with only 1-2 relevant sources, the agent will stretch to fill 3 findings — degrading signal-to-noise. When the topic is well-documented (authentication, caching, API design), 8 findings is artificially restrictive.

The defender was right that over-constraining search strategy is bad. But they then over-constrained output volume, which is the same mistake applied to the other end.

**Better approach**: Instead of count bounds, specify a **relevance filter**: "Include only findings that would change a design decision. If only one finding meets this bar, one finding is correct output." This keeps quality high regardless of volume.

---

## B. New Vulnerabilities in Proposed Changes

### B1. The Complexity Triage Step (Step 0) Creates a Chicken-and-Egg Problem

The defender proposed — and presented as their headline insight — a Step 0: "Read idea.md, classify complexity as Small/Medium/Large, and scale subsequent research accordingly."

**What breaks**: The agent hasn't done any research yet. It's classifying complexity based on a raw idea description. A two-sentence idea.md could describe something trivial ("add a favicon") or something deceptively complex ("add SSO support" — which sounds simple but involves SAML, OIDC, session management, and security review). The agent cannot reliably assess complexity from the idea alone because **discovering complexity IS the research**.

If the agent classifies "add SSO" as Small and scales down its research, it misses the security implications entirely. The downstream architect gets thin research for a high-risk feature. This is worse than the current spec, which at least applies uniform depth.

**If this is added at all**, it should be a post-research step that adjusts the output's emphasis, not a pre-research step that gates input. Or it should default to Medium and only downscale to Small when idea.md explicitly contains phrases like "minor fix," "config change," or "cosmetic update."

---

### B2. The Codebase Exploration Cap (15-20 Files) Creates a False Ceiling

**Defender's fix**: "Inspect up to 15-20 files maximum."

This is the same class of error as the web research output bounds. On a small project with 12 files total, 15-20 is more than the entire codebase — the cap is meaningless. On a large monorepo, the 20 most important files might all be configuration, and the actual implementation patterns live in file 21-30.

More critically: the defender said "Begin with high-signal files: README, main entry points, configuration files." That's 5-8 files consumed on project meta-information, leaving only 7-12 slots for actual code pattern discovery. In a project with 6 microservices, that's 1-2 files per service — not enough to identify patterns.

**Better approach**: Don't cap by count. Cap by objective: "Stop exploring when you can answer: (1) what patterns does the codebase use for [relevant concern from idea.md]? (2) are there existing implementations that overlap with the proposed feature? (3) what conventions must new code follow? If you cannot answer these after 20 files, note the gaps as open questions rather than continuing to explore."

Objective-based termination is smarter than count-based termination for an LLM agent that can actually evaluate whether it has enough information.

## C. Insufficient Rebuttals

### C1. The Stack Constraint Defense for Quality Commands Is Circular

The defender rejected Point 3 by citing Principle 8 ("Node.js stack"). Two problems:

**First**: The spec itself contains no stack restriction. It accepts any `idea.md` and processes it. If someone feeds it a Python project's idea.md, nothing in the spec stops execution or warns the user. The defender is relying on an **external constraint the spec doesn't enforce**. A spec that silently produces wrong output for valid-but-unexpected input is buggy, even if the expected usage is narrower.

**Second**: The defender claims "CI workflow check catches quality commands for ANY language." This is only true if the project HAS CI workflows. Many early-stage projects — exactly the kind that need a spec-driven research phase — don't have `.github/workflows/` yet. For those projects, the researcher checks `package.json` (doesn't exist for Python), `Makefile` (maybe), and CI (doesn't exist). All three return nothing. ### C2. The Rejection of Feasibility Anchoring Conflates Precision with Usefulness

The defender argued that T-shirt sizing is "intentionally relative" and that anchoring adds "false precision." This confuses two things: **who the estimate is for** and **how precise it needs to be**.

The defender says: "What matters is the relative ordering: if A is S and B is XL, the architect knows where to focus."

**Failure scenario that survives this defense**: The researcher estimates "Authentication: M" and "Database schema: M." They're the same size. The architect allocates equal design effort. But the researcher was thinking "M for an experienced developer" for auth and "M for this specific simple schema" for the database. In reality, auth requires OIDC integration (a week of work) and the schema is 3 tables (a few hours). The T-shirt sizes are equal, but the actual work isn't, because "M" meant different things in the researcher's implicit reasoning.

The fix isn't false precision — it's **consistency**. "Estimate relative to a single developer working within this project's stack" is not a precise anchor. It's a normalization instruction. Without it, the agent applies different implicit baselines to different components within the same table, and the relative ordering — which the defender says is what matters — becomes unreliable.

One sentence fixes this: "All estimates assume a developer familiar with the project's stack working within this codebase." That's not over-engineering. It's preventing the one failure mode the defender claimed mattered (relative ordering) from breaking.

### C3. The Rejection of Append-Only Concerns Ignores the Actual Consumer

The defender argued that 20 iterations are unrealistic and that the file is scoped to a single spec. Fine. But the defender's own proposed fix for idempotency EXPLICITLY allows re-runs:

> "Use --force to re-run."

So the defender simultaneously argues (a) re-runs rarely happen and (b) we should add explicit support for re-runs. If `--force` re-runs exist, then `.progress.md` will accumulate contradictory learnings from multiple research passes. Not 20 — maybe 2 or 3. But that's enough.

Research pass 1 discovers "Library X supports our use case." Human reviews, says "also check Library Y." Force re-run. Research pass 2 discovers "Library Y is superior to Library X." Now `.progress.md` contains both learnings with equal weight. The downstream product-manager reads both and has to guess which is current.

The fix I proposed in Round 1 (timestamps per entry) costs nothing and resolves this entirely. The defender's objection ("20 iterations unrealistic") doesn't apply to 2-3 iterations, which the defender's own changes enable.
