## Round 1 — CRITIC

# Critique: `spec-executor` Agent Definition

**Content type:** Agent system prompt / behavioral specification for an autonomous code executor in a multi-agent pipeline.
**"Improved" means:** Fewer ambiguous instructions that cause the agent to stall, corrupt state, or produce silent failures. Tighter contracts between this agent and its coordinator. Elimination of race conditions and data loss scenarios.

---

## Critical Issues

### 1. The `flock` concurrency model is broken on multiple levels

**Section:** Parallel Execution

**The `sed -i` approach to updating `tasks.md` is catastrophically fragile.** The lock protects the write, but `sed -i` does a read-modify-write on the *entire file*. If two executors lock sequentially, the second one reads the file the first just wrote — that works. But the instruction says to also `git add` tasks.md inside a *separate* lock block (`.git-commit.lock`). Between releasing `.tasks.lock` and acquiring `.git-commit.lock`, another executor can modify `tasks.md` again. The committed version may contain changes from a task that hasn't been committed yet, or miss changes from one that has.

**Failure scenario:** Executor A marks task 2.1 done, releases `.tasks.lock`, waits for `.git-commit.lock`. Executor B marks task 2.2 done, releases `.tasks.lock`, acquires `.git-commit.lock` first, commits tasks.md showing both 2.1 and 2.2 done. Executor A then commits tasks.md showing only 2.1 done (stale read from before B's edit). Git won't catch this — it's not a merge conflict, it's a clean overwrite. Task 2.2 completion is erased from history.

**Fix:** Use a single lock that spans both the `sed` update and the `git add && git commit`. Or better: don't use `sed -i` — read the file inside the lock, modify in memory, write back, stage, and commit all under one lock. The two-lock design is a footgun.

---

### 2. Step 6 (Update progress) happens AFTER Step 5 (Commit)

**Section:** Execution Flow, Steps 5-6

The agent commits first, *then* marks the task as `[x]` in `tasks.md` and updates the progress file. But Step 5 explicitly stages `tasks.md` and the progress file in the commit. So either:

- (a) The commit in Step 5 includes an *un-updated* tasks.md (task still marked `[ ]`) and stale progress file, which is wrong, OR
- (b) The agent is expected to intuit that it should update these files before committing despite the step ordering saying otherwise.

**Failure scenario:** The agent follows instructions literally. It commits with tasks.md still showing `[ ]` for the current task. The next agent reads `.progress.md` from the commit and sees no record of completion. The task gets re-executed. Duplicate work, possible conflicts.

**Fix:** Reorder explicitly: Step 5 = Update progress (mark `[x]`, write progress), Step 6 = Commit (stage everything including updated tracking files), Step 7 = Signal.

---

### 3. No rollback on partial failure

**Section:** Execution Flow (entire)

If the agent completes Step 3 (Implement) — modifying files on disk — but Step 4 (Verify) fails all 3 attempts, the document says to "document the failure in .progress.md and do NOT signal completion." But the *modified files remain on disk and unstaged*. The working tree is now dirty with a half-implemented task. The next executor or the coordinator inherits a corrupted workspace.

**Failure scenario:** Task 2.1 modifies `src/api.ts` and `src/db.ts`. Verification fails. Task 2.2 runs next, also touches `src/api.ts`. It reads the file and sees the broken partial implementation from 2.1, not the clean baseline. Task 2.2 builds on a broken foundation. Everything downstream is poisoned.

**Fix:** Add an explicit rollback step: `git checkout -- <files from Files section>` when verification fails permanently. Or at minimum, mandate that the failure documentation in `.progress.md` lists every file modified so the coordinator can clean up.

---

### 4. "Modify ONLY files listed in the Files section" is unenforceable and contradictory

**Section:** Constraints + Step 5

The constraint says modify ONLY listed files. But Step 5 also stages `tasks.md` and the progress file — those are modifications to files NOT listed in the task's Files section. More importantly: many real implementation tasks require modifying a `package.json` for dependencies, or a config file for a new module. If the task's Files list is incomplete (which happens constantly in spec-driven workflows), the agent is instructed to either violate the constraint or produce broken code that can't pass verification.

**Failure scenario:** Task says to create `src/new-module.ts` (listed in Files) that imports a new npm package. The agent can't run `npm install` because `package.json` and `package-lock.json` aren't in the Files list. Verification fails. The agent burns 3 retries, documents failure, and halts — all because of an incomplete Files list in the task definition.

**Fix:** Add an escape hatch: "If verification fails because a file outside the Files list needs modification, document the additional file in progress, add it to the staged files, and proceed. Flag this as a deviation." Rigid adherence to an incomplete spec is worse than documented deviation.

### 5. The `[VERIFY]` delegation has no error contract

**Section:** Step 2 — Check for [VERIFY] tasks

The agent is told to "delegate to the `qa-engineer` agent" but there's zero specification of:
- How to invoke that agent (what tool? what parameters?)
- What to do if the QA agent fails or times out
- What to do if the QA agent returns an ambiguous result
- Whether to still signal `TASK_COMPLETE` after delegation
- Whether the QA agent handles its own progress updates or the executor does

**Failure scenario:** The executor encounters a `[VERIFY]` task. It spawns the QA agent. The QA agent crashes or returns without a clear pass/fail signal. The executor has no fallback behavior defined. It either hangs, signals completion incorrectly, or silently drops the task. The coordinator sees no progress and no failure.

**Fix:** Define the delegation contract explicitly: invocation mechanism, expected return values, timeout handling, and who owns the progress update. "Delegate to X" without a protocol is a wish, not an instruction.

### 6. Three-retry loop has no backoff or diagnostic differentiation

**Section:** Step 4 — Verify (mandatory block)

"If it fails, diagnose the issue, fix it, and re-run. Repeat up to 3 times." This treats all failures identically. A compilation error, a flaky network call, and a fundamentally wrong approach all get the same treatment: fix and retry. But "fix" for a flaky test might be re-running; "fix" for a logic error requires rethinking; and "fix" for an environment issue (missing binary, wrong Node version) is impossible within the executor's scope.

**Failure scenario:** The verify command fails because the test environment is misconfigured (wrong DB path, missing env var). The agent "fixes" the code 3 times, making it progressively worse, then gives up. The real fix was setting an environment variable, which is outside its scope. Three wasted iterations that damaged working code.

**Fix:** Classify failures. If the verify command's stderr indicates an environment/infra issue (command not found, permission denied, connection refused), fail fast with a clear blocker note instead of burning retries on unfixable problems.

### 7. No file-level conflict detection before writing

**Section:** Step 3 — Implement

The agent modifies files listed in the task. In parallel execution, two executors might be assigned tasks that touch different files — but nobody checks. The task definitions might have overlapping Files lists due to a planning error. There's no guard against this.

**Failure scenario:** Tasks 2.1 and 2.2 both list `src/config.ts` in their Files section (planner mistake). Both executors modify it concurrently. The `flock` on git commit serializes the commits, but the second commit overwrites the first's changes to that file. No merge, no conflict detection, just silent data loss.

**Fix:** Before starting implementation, check if any file in the Files list is locked by another executor (e.g., a `.file-locks/` directory with per-file lock files). If a conflict is detected, halt immediately and report to the coordinator.

---

### 8. Phase-specific behavior is dangerously vague

**Section:** Phase-Specific Behavior

"Phase 1: Skip tests, accept hardcoded values, move fast." But the agent receives a `taskBlock` — **how does it know which phase it's in?** There's no input parameter for phase. There's no instruction to infer it from the task ID numbering (1.x = Phase 1?). There's no fallback.

**Failure scenario:** A Phase 3 testing task arrives. The agent has no phase signal, defaults to Phase 1 behavior ("move fast, skip tests"), and produces a task that skips the very tests it was supposed to write. It passes its own verify (which only checks compilation), commits, and signals completion. The testing phase is hollow.

**Fix:** Either pass `phase` as an explicit input parameter, or define unambiguous rules for inferring phase from the task block content. "Phase-specific behavior" without a phase identifier is dead code.

---

### 9. The "Source of Truth" conflict resolution is a deadlock

**Section:** Source of Truth

"If the task block conflicts with another file you read later, stop and record the conflict in progress instead of guessing." This sounds safe but creates a silent deadlock. The agent stops, writes to progress, doesn't signal completion. The coordinator sees an incomplete task with no `TASK_COMPLETE`. If the coordinator doesn't actively poll progress files for conflict notes (and there's no indication it does), this task is stuck forever.

**Fix:** Define an explicit conflict signal — a dedicated output like `TASK_BLOCKED: conflict between taskBlock and <file>` — that the coordinator is contractually required to handle. Writing to a progress file that nobody reads is not communication.

---

### 10. Commit message rigidity prevents useful metadata

**Section:** Step 5 — Commit (mandatory block)

"Commit with the EXACT message from the task's Commit line. Do not modify, abbreviate, or rephrase it." This prevents appending useful metadata like the spec name, task ID, or executor run ID. In a multi-spec, multi-executor environment, git log becomes a wall of context-free messages like "Add user validation" with no way to trace which spec or task produced them.

**Fix:** Allow appending a standardized trailer (e.g., `Spec: P227, Task: 2.1`) to the commit body while keeping the subject line exact. This preserves traceability without violating the intent of consistent messaging.

## Round 1 — DEFENDER

# Defense: `spec-executor` Agent Definition

## What the Critic Missed

Before addressing the ten points: **no, the critic did not focus on the most important issue.**

The most critical concern about this agent prompt — one that affects *every single task execution*, not just edge cases — is the **lack of structured guidance for when and how to consume broader context.** The mandatory block states:

> Fresh context: you receive ONLY the task block and .progress.md content. You do NOT receive research.md, requirements.md, design.md, or other task blocks. If you need information from other files, use the Read tool to fetch them explicitly.

This creates a paradox: the executor must produce code that fits into a broader system design, but it receives *zero architectural context by default* and has no heuristic for *when* reading design.md or requirements.md is necessary versus optional. A task that says "Do: Create the authentication middleware" can technically pass verification (responds 200 OK) while being architecturally wrong (uses sessions when the design specifies JWT, or stores tokens in-memory when the design specifies Redis).

This matters more than every issue the critic raised because:
- It affects **100% of non-trivial tasks**, not just parallel execution or failure paths
- It produces **silently wrong code** that passes verification — the hardest failures to catch
- It compounds across tasks — each context-blind executor drifts further from the design

**Fix:** Add a step between Parse and Implement: "Step 2.5: Context check. If any Do step involves creating a new module, interface, or integration point, read `{basePath}/design.md` to verify alignment. If the task modifies an existing file, read that file first to understand conventions. Record any design context consulted in progress."

Now, to the critic's points.

## Point 1: The `flock` concurrency model — REJECT

The critic's catastrophic scenario is **wrong on the mechanics of `git add`.**

Let me trace the actual execution:

1. Executor A acquires `.tasks.lock`, runs `sed -i` to mark 2.1 done, releases lock
2. Executor B acquires `.tasks.lock`, runs `sed -i` to mark 2.2 done (file now has BOTH marked), releases lock
3. Executor B acquires `.git-commit.lock`, runs `git add tasks.md` — **stages the current working tree file, which has both 2.1 and 2.2 marked** — commits, releases lock
4. Executor A acquires `.git-commit.lock`, runs `git add tasks.md` — **stages the same file, still showing both marked** — commits, releases lock

The critic claims "Executor A then commits tasks.md showing only 2.1 done (stale read from before B's edit)." This is false. `git add` does not cache the file at some earlier point — it reads the file *at the moment it's called*. Since both `sed` operations completed before either `git add` runs (serialized by `.tasks.lock`), both `git add` calls see the fully-updated file.

There is no "stale read." There is no silent data loss. Task 2.2's completion is NOT erased. The critic built an elaborate failure scenario on a misunderstanding of how `git add` works.

The only legitimate concern here is a **cosmetic one**: Executor A's commit diff might show task 2.2 marked as done, which is misleading in git blame. That's a minor auditability issue, not a "Critical" data loss bug.

**Verdict:** The two-lock design works correctly for its stated purpose. The critic's "Critical" rating is unwarranted.

---

## Point 2: Step ordering (commit before progress update) — PARTIAL CONCEDE

The critic correctly identifies that the step *numbering* implies updating progress after committing, which would mean the commit contains stale tracking files.

However, the critic is treating this as a specification for a bash script. **This is a prompt for an LLM agent.** LLM agents understand intent, not just sequence numbers. Any competent model reading "Step 5: Commit... stage tasks.md and progress file" and "Step 6: Mark task as [x] in tasks.md" would understand it needs to update the files before staging them. The steps describe *what to do*, and the model infers the logical dependency.

That said, **I concede the ordering is confusing and should be fixed** — not because an LLM would misexecute, but because clarity costs nothing and ambiguity invites edge-case failures across different model capabilities. The fix is trivial: swap Steps 5 and 6, rename accordingly. I accept the critic's proposed reordering.

---

## Point 3: No rollback on partial failure — PARTIAL CONCEDE

The failure scenario is real: modified files remain on disk after verification failure. But the critic's conclusion — "everything downstream is poisoned" — overstates the impact based on an assumption about how the coordinator works.

In a well-designed coordinator (which this document's contract implies):
1. The absence of `TASK_COMPLETE` signals failure to the coordinator
2. The coordinator reads `.progress.md`, sees the failure documentation
3. The coordinator runs `git checkout -- <files>` before dispatching the next task

The executor's job is to **implement and verify one task**. Workspace cleanup after failure is the coordinator's responsibility, just as garbage collection isn't the function's job — it's the runtime's.

**However**, I concede that the document should make the executor's failure obligations explicit. Not a full rollback (that adds complexity and failure modes to an already-failed path), but a mandatory inventory:

**Fix I'd accept:** After 3 failed verifications, append to progress: "Files modified: [list]. Working tree is dirty. Coordinator must reset these files before next task." This gives the coordinator exactly what it needs without making the executor responsible for cleanup it might botch.

## Point 4: Files-list constraint too rigid — REJECT

The critic frames this as a flaw in the executor, but it's **a deliberate architectural choice** that the critic is misidentifying as a bug.

The executor is intentionally constrained. The task planner is responsible for listing ALL files that need modification. If `package.json` needs changing, it should be in the Files list. If it's not, that's a **planning bug**, and the correct response is exactly what the document prescribes: fail, document, don't signal completion.

The critic's proposed fix — "document the deviation and proceed" — would create a system where:
- The executor silently modifies files the planner didn't account for
- Other parallel executors don't know about these modifications
- The coordinator's mental model of what each task touched is wrong
- Debugging becomes a nightmare because the actual scope exceeded the planned scope

The document already has the escape hatch: "If truly blocked after exhausting all tools, document the blocker in .progress.md Learnings and do NOT output TASK_COMPLETE." This IS the correct behavior. The planner gets the feedback ("task 2.1 blocked: package.json not in Files list"), fixes the task definition, and re-dispatches.

**Rigid adherence to a spec you didn't write is a feature of a trustworthy executor, not a bug.** If the executor starts "helpfully" expanding its scope, you've lost the ability to reason about what any given task will do.

## Point 5: [VERIFY] delegation has no contract — PARTIAL CONCEDE

The critic is right that the delegation is underspecified. "Delegate to the `qa-engineer` agent" without invocation details, timeout handling, or return value expectations is incomplete.

However, the critic overstates this by calling it a "silent task drop" risk. In the orchestration framework this runs in (Claude Code's Agent tool), invocation follows a standard pattern — the executor would use the Agent tool with `subagent_type: "qa-engineer"`. The framework handles crashes and timeouts at the infrastructure level.

**I concede** that the document should specify:
- The executor waits for the QA agent to return
- If the QA agent returns VERIFICATION_PASS → proceed to commit/signal
- If the QA agent returns VERIFICATION_FAIL → document in progress, do NOT signal completion
- The QA agent owns its own progress updates for verify-specific notes

I'd add a paragraph to Step 2 with these four bullets. The invocation mechanism itself is framework-specific and shouldn't be hardcoded in the prompt.

## Point 6: Three-retry loop doesn't classify failures — PARTIAL CONCEDE

The critic's scenario — "the agent 'fixes' the code 3 times, making it progressively worse" — assumes the agent is mindlessly patching code. But this is an LLM agent. When it sees `bash: jq: command not found`, it doesn't modify source code. It recognizes this as an environment issue. "Diagnose the issue" is literally what LLMs are good at.

That said, the critic's core suggestion is sound: **fail fast on clearly environmental failures.** Adding a classification heuristic doesn't cost much and prevents the rare case where a less capable model burns retries on unfixable problems.

**Fix I'd accept:** Add to Step 4: "Before retrying, classify the failure. If the verify error indicates a missing tool, permission issue, or environment misconfiguration (not a code logic error), document the blocker immediately and do NOT retry — the fix is outside executor scope."

This is lighter than the critic's proposal and captures the real value.

## Point 7: No file-level conflict detection — REJECT

Same architectural argument as Point 4. **File assignment is the planner's responsibility.** If two parallel tasks list the same file, the planner has a bug. Adding a file-locking system to the executor:

- Adds significant complexity (lock files, deadlock potential, cleanup on crash)
- Masks planning bugs instead of surfacing them
- Creates a new failure mode (lock contention, stale locks from crashed executors)
- Solves a problem that the coordinator/planner should prevent by construction

The critic provides no evidence that overlapping Files lists are common in practice — they say "which happens constantly in spec-driven workflows" without substantiation. In a well-structured task planner (which this system includes), parallel tasks are specifically designed to touch disjoint files. That's the whole point of the parallel batch design.

If we're worried about planner bugs, the right fix is a **pre-dispatch validation in the coordinator** that checks for file overlaps before launching parallel executors — not runtime locking in each executor.

## Point 8: Phase is never communicated — PARTIAL CONCEDE

The critic is right that there's no explicit `phase` parameter. But they're wrong that the agent has "no way to know."

The task numbering convention (1.x = POC, 2.x = Refactoring, 3.x = Testing, etc.) is established in the task planner's output format. And task content is inherently phase-revealing: a task that says "Write unit tests for X" is obviously Phase 3 regardless of numbering.

**However**, relying on implicit inference when an explicit parameter is cheap is poor design. I concede this point partially.

**Fix I'd accept:** Add `phase` as an optional input parameter: "If `phase` is provided, apply phase-specific behavior. If not, infer from task numbering (1.x = Phase 1, 2.x = Phase 2, etc.)." This gives the coordinator a clean override while maintaining backward compatibility.

---

## Point 9: Conflict "stop" has no escalation path — REJECT

The critic says writing to progress is "not communication" because "nobody reads" the progress file. This is wrong — the entire coordinator-executor contract is built around progress files. The coordinator dispatches a task, waits for `TASK_COMPLETE`, and if it doesn't arrive, reads the progress file for the reason. That's the communication protocol.

The absence of `TASK_COMPLETE` IS the escalation signal. The progress file IS the detail channel. The critic is proposing a redundant signal (`TASK_BLOCKED`) that duplicates information already available through the existing protocol.

Adding a separate `TASK_BLOCKED` output is fine as a convenience — I wouldn't oppose it — but calling the current design a "permanent deadlock" is wrong. The coordinator polls for completion and reads progress on failure. That's the contract.

## Point 10: Commit message rigidity — REJECT

The critic wants metadata appended to commit messages. But the commit message comes from the **task block**, which is authored by the **task planner**. If the planner wants to include `Spec: P227, Task: 2.1` in the commit message, they put it in the Commit line. The executor faithfully reproducing what the planner specified is correct behavior.

The critic's fix would have the executor unilaterally adding metadata the planner didn't specify. This:
- Violates the executor's role as a faithful implementer
- Creates inconsistency if different executors append different metadata formats
- Mixes concerns — traceability should be the planner's decision, not the executor's

If traceability is important (and it is), **fix the task planner to include it in the Commit line.** Don't have every executor independently deciding what metadata to append.

## A. The Defender's Own Fix Creates a New Deadlock Vector

The defender proposes adding "Step 2.5: Context check — read `design.md` if creating a new module or integration point."

**The defender proposes a fix for context blindness that is incompatible with the existing conflict-halt rule, and didn't notice.** Either the conflict rule needs to be softened (e.g., "informational discrepancies are noted but do not halt execution; only structural contradictions halt") or the context check needs to distinguish "stale but non-blocking" from "fundamentally incompatible." ## D. Commit Failure Inside `flock` Is Completely Unhandled

**NEW vulnerability.** The locked git block:

```bash
(
  flock -x 200
  git add <files>
  git commit -m "<message>"
) 200>"{basePath}/.git-commit.lock"
```

What happens when `git commit` fails? Causes include:
- Pre-commit hook rejects (lint failure, secret detection — the user's own `git-secrets-check.sh`)
- Index lock contention (`index.lock` exists from a crashed prior operation)
- Disk full
- Merge conflict in the index (another process modified the index outside the lock)

At this point in the flow, the executor has already:
1. Modified implementation files on disk ✓
2. Updated `tasks.md` to `[x]` (per corrected step ordering) ✓
3. Written to the progress file ✓

But the commit didn't happen. The tracking files say "done." The git history says "never happened." The next executor reads `tasks.md`, sees the task marked complete, skips it. **The task is permanently marked complete without ever being committed.** This is worse than the step-ordering bug from Point 2 — it's a silent phantom completion.

The document has no error handling for commit failure at all. Not in the main flow, not in the parallel flow.

**Fix:** After `git commit`, check the exit code. If non-zero: revert the `tasks.md` mark to `[ ]`, remove the completion entry from progress, and either retry the commit or fail the task entirely. The tracking state must be consistent with the git state.

## E. The File Inventory Fix for Point 3 Is Incomplete

The defender's accepted fix for verification failure: "append to progress: 'Files modified: [list].'"

The executor runs bash commands during Step 3 (Implement). These commands can modify files *beyond* those in the Files list through side effects:
- `npm install` modifies `package-lock.json` and `node_modules/`
- A database migration modifies a `.db` file
- A build step generates artifacts in `dist/` or `.next/`
- `npx` auto-installs a package, modifying the global cache

The executor doesn't track every file touched by every bash command it runs. It only knows about files it explicitly wrote via `Edit`/`Write` tools. The inventory will list the intentional modifications and miss the side effects. The coordinator, relying on this inventory for cleanup, will `git checkout` the listed files and leave the side-effect modifications in place.

**Fix:** Instead of an executor-assembled inventory (which is inherently incomplete), the failure path should run `git diff --name-only` and `git status --porcelain` to capture the *actual* working tree state. Report the output verbatim. The filesystem is the source of truth for what changed, not the executor's recollection.

## F. The Defender's Rebuttal on Point 1 Is Logically Wrong (Even If the Conclusion Survives)

The defender claims:

> Since both sed operations completed before either git add runs (serialized by .tasks.lock), both git add calls see the fully-updated file.

This ordering is NOT guaranteed. The two locks (`.tasks.lock` and `.git-commit.lock`) are independent. A valid interleaving:

1. **A** acquires `.tasks.lock`, seds 2.1 done, releases
2. **A** acquires `.git-commit.lock`, `git add tasks.md` ← file has only 2.1 marked, `git commit`, releases
3. **B** acquires `.tasks.lock`, seds 2.2 done, releases
4. **B** acquires `.git-commit.lock`, `git add tasks.md` ← file has both marked, `git commit`, releases

In this interleaving, A's commit contains `tasks.md` with only 2.1 done. B's commit contains both. There is no permanent data loss (B's commit has the full state), so the original critic's "Critical" rating was indeed too high. But the defender's *reasoning* — "both seds complete before either git add" — is wrong. It assumes an interleaving that nothing enforces.

This matters because: if A's commit records only 2.1 done, and a process inspects A's *specific commit* (e.g., a CI pipeline triggered per-commit, a bisect operation, a coordinator that reads tasks.md at a specific commit SHA), it sees an incomplete picture. The defender dismissed this as "cosmetic." It's not cosmetic if any tooling operates on individual commits rather than HEAD.

I'm not re-raising Point 1 — I'm pointing out that the defender's defense rests on a false premise about execution ordering, and the dismissal of the intermediate-state problem underestimates how git history is actually consumed.

## G. Worktree Isolation Breaks the Entire Concurrency Model

**NEW vulnerability.** The Agent tool in this framework supports `isolation: "worktree"`, which runs the agent in a temporary git worktree — a separate working directory with its own copy of the files.

If the coordinator launches parallel executors with worktree isolation:
- Each executor has its own copy of `tasks.md` and `.progress.md`
- The `flock` lock files use `{basePath}`, which resolves to different physical paths in each worktree
- **The locks don't actually interlock.** Each executor locks its own copy. No mutual exclusion occurs.
- `sed -i` modifies a worktree-local copy. `git commit` creates a commit in the worktree's branch. Merging these branches back is an entirely unspecified operation.

The document's entire parallel execution model assumes a shared filesystem. It never states this assumption. If a coordinator launches executors in worktrees (which the framework explicitly supports), every concurrency guarantee evaporates.

**Fix:** Add an explicit precondition: "Parallel execution requires a shared working directory. Do NOT use worktree isolation for parallel task batches." Or redesign the concurrency model to work across worktrees (branch-per-task with coordinator-managed merge).
