## Round 1 -- CRITIC

# Ruthless Critique: `researcher` Agent Prompt

## PHASE 1 — FRAMING

**Document type:** Agent system prompt / instruction set for a "researcher" agent within a multi-agent spec-driven development pipeline. It defines the agent's role, execution steps, inputs, and output contract.

**"Improved" means:** Instructions that produce consistent, reliable output regardless of which LLM or CLI runtime executes them; robust error handling; clear contracts with upstream (idea.md) and downstream (product-manager, architect) agents; no silent failure paths.

## PHASE 2 — ANALYSIS (ordered by damage potential)

### 1. The agent has no idea where the codebase is

**Section:** "When Invoked" + "Step 2: Codebase Exploration"

`basePath` is defined as `~/spec-drive-projects/P###-name/spec/` — the spec directory. Step 2 says "Explore the target codebase." But there is **no parameter for the codebase root.** The agent receives a spec folder and is told to `Glob: **/*.ts` — from where? The spec directory? That's just markdown. The project root? It's undefined.

**Failure scenario:** The agent runs `Glob` from `basePath`, finds zero source files, produces a "Codebase Analysis" section that says "no code found," and the product-manager proceeds assuming greenfield when there's actually an existing codebase at `~/claw-projects/some-project/`. Every architectural decision downstream is uninformed.

**Fix:** Add a `codebasePath` parameter (explicit) or define a deterministic resolution rule (e.g., "look for a `codebase_root` field in `.spec-drive-state.json`"). Never make the agent guess.

---

### 2. `WebSearch` / `WebFetch` / `Glob` / `Grep` / `Read` are fictional tools

**Section:** Steps 1, 2, 3

These are pseudo-commands that don't map to any real tool in Claude CLI, Codex CLI, or Gemini CLI. The agent has `model: inherit`, meaning it could be any backend. Claude CLI has no web search. Codex CLI has no web search. The `Glob`/`Grep`/`Read` syntax doesn't correspond to any MCP tool or built-in.

The document mandates: "NEVER skip web search. External information may reveal blockers invisible in the codebase." This is a **hard constraint the agent cannot fulfill** in most execution environments.

**Failure scenario:** The agent either (a) hallucinates web search results to satisfy the mandatory constraint, producing fabricated URLs and fictitious "best practices," or (b) reports it cannot perform web search and stops, blocking the pipeline. Both outcomes are bad. Option (a) is worse because it's silent.

**Fix:** Map each pseudo-command to a concrete tool name per CLI backend, or define an explicit fallback: "If web search is unavailable, state 'Web search not available in this environment' in the External Research section and proceed." Make the constraint conditional on capability.

---

### 3. State file manipulation is fragile and destructive

**Section:** "Final Step: Set Awaiting Approval"

```bash
jq '.awaitingApproval = true' {basePath}/.spec-drive-state.json > /tmp/sd-state.json && mv /tmp/sd-state.json {basePath}/.spec-drive-state.json
```

Three problems:
- **No existence check.** If `.spec-drive-state.json` doesn't exist, `jq` errors, the pipeline writes nothing to `/tmp/sd-state.json`, and the `&&` short-circuits. Silent failure — the pipeline never pauses for approval, and the product-manager charges ahead on unreviewed research.
- **Race condition.** If another agent writes to the state file between the `jq` read and the `mv`, those writes are lost.
- **`jq` not guaranteed.** It's an external dependency with no fallback.

**Failure scenario:** State file doesn't exist on first run of a new project. `jq` fails. No approval gate is set. The product-manager agent picks up unreviewed research immediately. Garbage in, garbage out through the entire pipeline.

**Fix:** Check existence first. Create with sensible defaults if missing. Use atomic write (e.g., `sponge` or `mktemp` in the same filesystem to avoid cross-device `mv` issues). Add a verification read after write.

---

### 4. "Sole input" claim contradicts the actual execution

**Section:** "Input"

> "Read `{basePath}/idea.md`. This is your sole input."

The agent then proceeds to read package.json, Makefile, CI workflows, and arbitrarily many source files. These are independent inputs, not derivatives of idea.md.

**Failure scenario:** A literally-minded LLM interprets "sole input" as a scoping constraint and skips or de-prioritizes codebase exploration. The Codebase Analysis section becomes a perfunctory afterthought. This is especially likely with smaller/cheaper models in the inherit chain.

**Fix:** Say "idea.md is your **primary directive** — it defines what to research. The codebase and web are your **evidence sources**."

---

### 5. No greenfield project handling

**Section:** Steps 2, 3

Every instruction in Steps 2 and 3 assumes existing code: "find relevant files," "locate related code," "check package.json scripts." For a brand-new project with no codebase, the agent has no guidance.

**Failure scenario:** Research for a new project produces "Not found" for every codebase question and every quality command. The Feasibility Assessment has no grounding. Downstream agents receive a research document that's 60% "Not found" and make decisions blind.

**Fix:** Add explicit handling: "If no codebase exists yet, state this clearly and focus on: (a) identifying a starter template or boilerplate, (b) recommending initial project structure based on the stack constraints in idea.md, (c) documenting quality commands to be *created* during implementation."

### 6. Zero security research in a security-paranoid system

**Section:** All steps — conspicuous absence

The user's principles include "Seguridad no es opcional," "assume breach," "don't trust external dependencies blindly." Yet the researcher agent has **no step for security analysis**: no dependency audit, no CVE checks, no threat modeling, no assessment of data sensitivity implications.

**Failure scenario:** Research for a feature involving, say, OAuth integration or file uploads recommends an npm package without checking its CVE history. The architect adopts it. A known vulnerability ships to production.

**Fix:** Add "Step 2.5: Security Scan" — check dependencies for known vulnerabilities, assess data flow implications, flag any external service trust boundaries. This should be mandatory, not optional, given the operator's security posture.

### 7. Feasibility estimates are groundless

**Section:** "Step 4: Feasibility Assessment"

The effort scale (S/M/L/XL) and risk ratings require knowledge of team velocity, skill composition, current workload, and organizational constraints. The agent has none of this. The only input is idea.md and whatever code it found.

**Failure scenario:** Agent rates a component as "M effort, Low risk" because the code pattern looks simple. In reality, it touches a shared module with three other projects, requires coordination with another team, and the only person who understands it is on vacation. The product-manager builds a timeline based on fiction.

**Fix:** Either (a) scope feasibility explicitly to "technical complexity assessment only — does not account for team capacity or organizational factors," or (b) provide a team context file as input (velocity, known bottlenecks, availability).

### 8. `.progress.md` append-only policy creates contradictions over time

**Section:** "Progress Update"

"Append only. Never delete existing learnings." If research is run multiple times (iteration, retry after failure, scope change), earlier learnings may directly contradict later ones. There's no reconciliation mechanism.

**Failure scenario:** First run appends "Library X v2 is stable and recommended." Scope changes, second run appends "Library X v2 has a critical bug, use v3." Both lines coexist. A downstream agent reads the file top-to-bottom and uses v2.

**Fix:** Append with timestamps AND status markers. Allow marking previous entries as `[SUPERSEDED]`. Or mandate that new entries that contradict previous ones must explicitly reference and override them.

### 9. No scope or token budget

**Section:** Steps 1 and 2

**Failure scenario:** Per the user's own memory: "vampire cron spec-research/researcher burned ~4.88M Opus tokens in 12h." With `model: inherit` potentially resolving to an expensive model, a vague idea.md triggers unlimited exploration.

### 10. Human approval bottleneck with no release valve

**Section:** "Final Step: Set Awaiting Approval"

"Research requires human sign-off before the product-manager proceeds. This is non-negotiable."

There's no timeout, no escalation, no notification mechanism. The pipeline hard-stops until a human notices.

**Failure scenario:** Research completes at 2 AM. Gab is asleep. The pipeline sits blocked for 8+ hours. For a system that values autonomous operation, this is a process bottleneck with zero mitigation. Multiply by several concurrent projects and the human becomes the permanent constraint.

**Fix:** Add a notification step (the system already has `notify-gab.sh`). Define an SLA: "If no response within N hours, send reminder." Don't remove the gate — but make it visible and tracked.

Only checks: package.json, Makefile, .github/workflows. Misses: `pyproject.toml`, `Cargo.toml`, `justfile`, `taskfile.yml`, `tox.ini`, `noxfile.py`, `pre-commit-config.yaml`, `docker-compose.yml` (for integration tests), `deno.json`, `go.mk`.

A template is provided, but there's no structural validation. The agent could omit the Quality Commands table, write prose instead of tables in Feasibility, or skip Open Questions entirely. Nothing catches this.

**Failure scenario:** The agent writes a freeform research document that looks plausible but is missing the Quality Commands section. The task-planner downstream expects structured data and either crashes or produces tasks with no verification steps.

**Fix:** Add a self-validation step before setting `awaitingApproval`: "Verify research.md contains all required sections: Executive Summary, External Research, Codebase Analysis, Quality Commands table, Feasibility Assessment table, Open Questions. If any section is missing, add it with explicit 'No findings' notation."

## Round 1 -- DEFENDER

# Defense of the `researcher` Agent Prompt

## The Concern the Critic Missed

Before addressing the twelve points: **the critic never questioned what happens when the agent fails mid-execution.**

This agent has five sequential steps, each producing intermediate knowledge, but there's no checkpoint mechanism. If the agent completes web research and codebase exploration (burning significant tokens) but then hits a context limit, timeout, or crash before writing `research.md`, *all work is lost*. Given the operator's environment — a Dell laptop with 16GB RAM, a 600-second watchdog timer, and documented history of orphaned `claude -p` processes from context compaction — this is not theoretical. It's the operational reality.

This matters more than most of the critic's points because it's a **silent total failure**: no output, no partial output, no error in the state file. The pipeline just... stalls. No `awaitingApproval` gets set, so from the pipeline's perspective, the agent is still "running." The human gets no notification, downstream agents get nothing, and the tokens are gone.

**Fix:** Write intermediate results progressively. After Step 1, write a partial `research.md` with just the External Research section. After Step 2, append Codebase Analysis. This way, a mid-flight failure leaves recoverable artifacts. Add to the instructions: *"Write each section to research.md as you complete it. A partial research.md is infinitely more valuable than none."*

Now, to the critic's points.

# Round 2 — CRITIC

## (a) Weak Fixes That Won't Actually Work

### The "Progressive Writes" Fix Introduces a Worse Problem Than It Solves

The defender's strongest contribution — the new issue about mid-execution crash recovery — proposes: *"Write each section to research.md as you complete it."*

This creates a **half-written research.md that passes the approval gate.**

Here's the kill chain:
1. Agent completes Steps 1-2, writes External Research and Codebase Analysis to `research.md`.
2. Agent crashes during Step 4 (Feasibility Assessment) — watchdog timeout, OOM, context compaction.
3. `research.md` exists on disk with two completed sections but no Feasibility Assessment, no Quality Commands, no Open Questions.
4. The agent never reaches the "Final Step" — `awaitingApproval` is never set. So the pipeline stalls, which is the *same* failure the defender was trying to fix.

Or worse: if the crash happens *after* the self-validation checklist (Point 12 fix) but *before* the state file write, you get a validated-looking `research.md` that's missing content from the sections the agent filled with "No findings for this section" as a placeholder, which is now indistinguishable from a genuine "no findings" result.

**The defender's fix for crash recovery directly conflicts with the defender's fix for output validation.** Progressive writes produce intermediate artifacts that look like final artifacts. The header checklist stamps them as complete.

**Actual fix:** Progressive writes must go to a *different file* — `research.partial.md` or `research.wip.md`. The final step atomically renames it to `research.md`. That way a crash leaves a clearly-labeled incomplete artifact, and the pipeline can distinguish "in progress" from "done." The state file should also track `currentStep` so a retry knows where to resume.

---

### The Token Budget Fix Has No Enforcement Mechanism

The defender proposed hard limits: "maximum 5 web queries, maximum 30 codebase files." But this is **text in a prompt, not a runtime constraint.** LLMs don't count their own tool invocations reliably. The defender criticized the critic for "asking it to self-assess when it's done enough exploring" and then proposed... asking it to self-assess when it's hit a count.

The 4.88M token vampire incident proves that prompt-level "please stop" instructions don't work against unbounded exploration loops. The agent that burned those tokens presumably had *some* instruction to be efficient. It didn't matter.

**Actual fix:** Token budgets must be enforced at the orchestration layer, not the prompt layer. The pipeline runner should kill the agent after N minutes or N tool calls. The prompt should *document* the intended limits (so the agent can prioritize), but enforcement belongs in the watchdog, not the willpower of the LLM.

---

### The Security Step Scoping Is Too Narrow

The defender conceded security research but scoped it to "check dependencies for CVEs" and "identify sensitive data flows." This is the *easy* part of security research and arguably the part that automated tools (npm audit, Snyk) already handle better than an LLM.

What's actually missing — and what a research agent is uniquely positioned to find — is:

- **Architecture-level security implications.** "This feature requires storing user credentials" → the researcher should flag this as a constraint for the architect, not wait for the architect to discover it.
- **Supply chain risk of proposed approaches.** If web research surfaces a library recommendation, the researcher should note its maintenance status (last commit, number of maintainers, bus factor), not just CVEs.
- **Threat surface expansion.** "Adding a webhook endpoint expands the attack surface from internal-only to internet-facing." This is a research finding, not a threat model.

The defender dismissed threat modeling as "an architect-phase activity." But identifying *that* threat modeling is needed *is* a research-phase finding. The researcher doesn't model threats; it flags that threats exist.

## (b) New Vulnerabilities Introduced by Proposed Changes

### The State File Resolution Chain Creates an Ambiguity Cascade

The defender proposed for Point 1:

> "Read `.spec-drive-state.json` for `codebasePath`. If absent, check idea.md for a project directory reference. If neither exists, report as greenfield."

This three-step resolution with a greenfield fallback means **any misconfiguration silently degrades to greenfield mode.** Typo in the state file path? Greenfield. `idea.md` mentions the project name but not the directory? Greenfield. State file exists but `codebasePath` key is misspelled as `codeBase`? Greenfield.

## Token Budget: Defense in Depth, Not "Enforcement Theater"

The critic argues prompt-level limits are "theater" because "LLMs don't count their own tool invocations reliably" and points to the 4.88M token incident.

The critic is creating a false dichotomy: prompt limits OR orchestration enforcement. The answer is **both layers**.

Let's be precise about the 4.88M incident. The operator's memory says: *"vampire cron `spec-research/researcher` burned ~4.88M Opus tokens in 12h."* That's a **cron job invoking the agent repeatedly over 12 hours** — not a single agent session ignoring its prompt constraints. The runaway cost came from *repeated invocation*, not from one session exploring unboundedly. The watchdog (already at 600 seconds) kills individual sessions. The cron scheduling was the actual bug.

Prompt-level limits serve a different purpose than orchestration kills: they give the agent **prioritization criteria**. "Maximum 5 web searches" means "you have 5 shots — pick the most important queries." Without this, even a watchdog-killed session wastes its entire budget on low-value exploration before being terminated.

The critic says *"The agent that burned those tokens presumably had some instruction to be efficient."* Presumably. But "be efficient" is vague. "Maximum 5 web queries, maximum 30 files" is specific. The difference between "try to be reasonable" and "you have a hard count" is the difference between a suggestion and a decision boundary.

Note: These limits are also enforced by the pipeline watchdog.

This makes both layers visible to the agent without pretending the prompt alone is sufficient.
