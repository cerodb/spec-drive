## Round 1 — CRITIC

# Critique: `researcher` Agent Prompt

**Content type**: Agent specification / operational prompt with embedded workflow logic.
**"Improved" means**: Unambiguous instructions that produce consistent, reliable research output across invocations, with no gaps that cause silent downstream failures.

---

## Critical Issues

### 1. The `jq` state-file update is a data-loss race condition

**Section**: "Final Step: Set Awaiting Approval"

```bash
jq '.awaitingApproval = true' {basePath}/.spec-drive-state.json > /tmp/sd-state.json && mv /tmp/sd-state.json {basePath}/.spec-drive-state.json
```

**Failure scenario**: If two agents or processes touch `.spec-drive-state.json` concurrently (e.g., a cron job updating progress, or the user running another spec command), the `jq` read and the `mv` write are not atomic. The second writer silently overwrites the first writer's changes. Worse: writing to the shared `/tmp/sd-state.json` path means two concurrent researcher invocations on different specs will clobber each other's temp file.

**Fix**: Use `mktemp` for the temp file (`jq '...' file > "$(mktemp)"`) or use `sponge` from moreutils. Better yet, define a shared state-update utility that uses file locking (`flock`).

---

### 2. No error handling when `.spec-drive-state.json` doesn't exist

**Section**: "Final Step: Set Awaiting Approval"

**Failure scenario**: On a fresh spec where the state file hasn't been initialized yet, `jq` reads a nonexistent file, outputs nothing, `mv` moves an empty file over the path, and the state is now an empty file — not valid JSON. Every downstream agent that reads this file will crash or behave unpredictably. The prompt says nothing about what to do if the state file is missing or malformed.

**Fix**: Add a guard: check the file exists and is valid JSON before patching. If missing, create it with a known schema. Define the expected schema somewhere.

---

### 3. `idea.md` "missing or empty" is the only input validation — what about malformed input?

**Section**: "Input"

**Failure scenario**: `idea.md` exists and is non-empty but contains only a title with no vision or constraints (e.g., `# My Project\n\nTBD`). The agent proceeds to research "TBD," performs meaningless web searches, and produces a research.md that looks complete but contains garbage. The downstream product-manager trusts it because it follows the template.

**Fix**: Define minimum viable content for `idea.md`. The agent should validate that it can extract at least a non-trivial vision statement. If extraction yields nothing actionable, halt — don't produce a hollow artifact.

---

### 4. Web search has no bounded retry or failure protocol

**Section**: "Step 1: Web Research"

The prompt says `NEVER skip web search` but provides zero guidance for when web search fails (network down, rate-limited, tool unavailable, results empty).

**Failure scenario**: WebSearch returns no results or errors out. The agent either (a) halts entirely and produces nothing, or (b) silently skips and writes "not found" for everything, producing a research.md that looks like a stub. Neither outcome is handled gracefully. The downstream agents have no signal to distinguish "research was thorough but found nothing" from "research tool was broken."

**Fix**: Define explicit behavior: if WebSearch fails, log the failure with the error, attempt N retries, and if all fail, mark the External Research section with a `⚠️ WEB SEARCH UNAVAILABLE` banner so downstream agents know the research is incomplete. Don't let this be silent.

---

### 5. Quality Command Discovery has no fallback for non-JS/non-Make projects

**Section**: "Step 3: Quality Command Discovery"

The three sources checked are: `package.json`, `Makefile`, `.github/workflows/*.yml`.

**Failure scenario**: A Python project with `pyproject.toml`, `setup.cfg`, or a Rust project with `Cargo.toml`, or a Go project with a plain `go.mk` — none of these are checked. The Quality Commands table will say "Not found" across the board, which is technically accurate but misleading. The task-planner will skip all verification steps for a project that actually has full CI.

**Fix**: Either (a) expand the discovery list to cover common ecosystems (`pyproject.toml`, `Cargo.toml`, `Justfile`, `Taskfile.yml`, `tox.ini`, `noxfile.py`), or (b) add a generic fallback step: grep the repo for common command patterns (`pytest`, `cargo test`, `go test`, `mix test`). The principle "missing commands are important" is correct — but incorrectly reporting commands as missing when they exist in a format you didn't check is worse than not checking at all.

---

### 6. Feasibility table has no calibration baseline — effort/risk ratings are pure vibes

**Section**: "Step 4: Feasibility Assessment"

The effort scale is defined (`S = hours, M = 1-2 days...`) but there's no guidance on what "risk" means, no risk framework, and no instruction on what inputs to use for calibration.

**Failure scenario**: The researcher agent rates everything "M effort, Low risk" because LLMs default to moderate answers. The architect takes these at face value. The project blows past timeline because an "M effort" component actually required integrating with an undocumented third-party API that needs auth negotiation. The risk assessment provided zero useful signal.

**Fix**: Define risk in terms of concrete factors: dependency on external systems (High), well-understood internal CRUD (Low), novel algorithm with no reference implementation (High). Give the agent a rubric, not just labels.

---

### 7. No deduplication or conflict check against prior research

**Section**: Entire flow

**Failure scenario**: The user re-runs research (the skill menu includes `ralph-specum:research` with "Run or re-run research phase"). The agent overwrites `research.md` entirely — discarding prior findings, including any human annotations or corrections added post-approval. The `.progress.md` append-only rule mitigates this slightly, but the main artifact is destroyed.

**Fix**: Before writing `research.md`, check if it already exists. If so, either (a) merge new findings with existing ones, (b) archive the old version (e.g., `research.md.bak-<timestamp>`), or (c) halt and ask for confirmation. The current design silently destroys prior work.

---

### 8. The `.progress.md` append is underspecified and will rot

**Section**: "Progress Update"

The instruction is "append discoveries under Learnings section." But there's no format for entries (no timestamps, no phase tags), no deduplication, and no size limit.

**Failure scenario**: After 5 re-runs of research, `.progress.md` has 50 redundant bullet points like "- Uses TypeScript" repeated across runs. The file becomes noise. Agents that read it later waste context window on duplicates.

**Fix**: Require timestamps on entries. Require a `[research]` phase tag. Before appending, grep for duplicates and skip them. Define a maximum learnings count or summarization strategy.

### 9. "Explore the target codebase" — but which codebase?

**Section**: "Step 2: Codebase Exploration"

The agent receives `basePath` (the spec directory, e.g., `~/spec-drive-projects/P###-name/spec/`). The codebase to explore is never explicitly defined. The glob patterns (`**/*.ts`, `**/*.js`) will run against... what? The spec directory? The parent? The current working directory?

**Failure scenario**: The agent globs from `basePath` (the spec folder) and finds only `idea.md` and `.spec-drive-state.json`. It reports "no existing codebase patterns found" and moves on. The actual codebase is three directories over. The research is useless for codebase analysis.

**Fix**: Either (a) require `idea.md` to specify a `codebasePath`, (b) add a `codebasePath` parameter to the agent invocation alongside `basePath`, or (c) define an explicit convention (e.g., `basePath/../../src/`). This is the single biggest gap in the prompt — an entire research step that may run against the wrong directory.

---

### 10. No output validation before writing

**Section**: "Output"

The agent is told to write `research.md` following a template. But there's no self-check: no instruction to verify all sections are populated, no minimum content thresholds, no validation that the YAML frontmatter is valid.

**Failure scenario**: The agent encounters an error mid-research (say, during codebase exploration), recovers partially, and writes a `research.md` with an empty "Codebase Analysis" section and a half-filled feasibility table. The file passes the "it exists" check. The product-manager proceeds with incomplete data. Nobody catches it until the design phase produces something unbuildable.

**Fix**: Add a self-validation step before writing: every section in the template must have content or an explicit `N/A — [reason]` marker. If any mandatory section is empty, halt and report instead of writing a partial artifact.

## Round 1 — DEFENDER

# Defense of the `researcher` Agent Prompt

## The Concern the Critic Missed

Before addressing the specific points: **the critic missed the most impactful gap in this document — the complete absence of confidence signals in the research output.**

The research.md template has findings, tables, and assessments, but no mechanism for the researcher to communicate *how confident* it is in each finding. The External Research section records "what, why relevant, adopt/adapt/ignore, source URL" — but not whether that finding is corroborated by five authoritative sources or tenuously inferred from a single tangential blog post. The Feasibility Assessment assigns effort/risk labels but doesn't disclose the evidential basis.

This matters more than most of the critic's points because it directly corrupts decision quality downstream. The product-manager and architect treat all findings as equally reliable. When the researcher says "Component X: Medium effort, Low risk" based on a single StackOverflow answer that may be outdated, and "Component Y: Large effort, High risk" based on three official docs and a known CVE — those two findings carry identical weight in the output format. The architect cannot triage. The human approver cannot efficiently audit. Every downstream decision inherits an invisible uncertainty that compounds.

**Proposed fix**: Add a `Confidence` column to key tables (e.g., `High/Med/Low` with mandatory one-line justification) and require each External Research finding to carry a `[N sources]` annotation. This is cheap, non-disruptive to the template, and gives downstream agents actual signal to work with.

## Responses to Each Point

### Point 1: `jq` race condition — **Partially concede, but severity is overstated**

The critic demonstrates a real failure with the hardcoded `/tmp/sd-state.json` path — two concurrent researcher invocations on different specs *would* clobber each other's temp file. That's a genuine bug. **I concede on the temp file path.**

However, the "race condition" framing overstates the risk. This agent operates within a sequential pipeline. The prompt itself says: *"Research requires human sign-off before the product-manager proceeds."* The entire workflow is phase-gated. There is no design scenario where two agents write to the *same* spec's state file concurrently — the human approval gate explicitly prevents this. A cron job updating progress would need to be written to touch this file, and nothing in the spec-driven workflow does that.

The critic meets criterion (a) — specific failure scenario with the temp file — and (b) — concurrent spec users affected. But criterion (c) — likelihood — fails for the race condition on a *single* spec's state file. Two researchers on *different* specs hitting `/tmp/sd-state.json` simultaneously is plausible; two writers on the *same* state file is not, given the workflow design.

**Fix I accept**: Replace `/tmp/sd-state.json` with a per-invocation temp file:
```bash
tmpfile=$(mktemp) && jq '.awaitingApproval = true' {basePath}/.spec-drive-state.json > "$tmpfile" && mv "$tmpfile" {basePath}/.spec-drive-state.json
```
No need for `flock` — that's overkill for a sequential pipeline.

The critic is right that a fresh spec could lack this file, and `jq` on a nonexistent file produces empty output. But the proposed fix ("define the expected schema somewhere") is vague.

This is self-contained, requires no external schema definition, and handles the empty-file case. The state file is a simple JSON bag — it doesn't need a formal schema; it needs a valid default.

The critic's scenario: `idea.md` contains `# My Project\n\nTBD` and the agent proceeds to research "TBD."

This misunderstands what the agent *is*. It's an LLM. It doesn't regex-match for a "vision" string — it reads the file and reasons about its content. If the file says "TBD," the agent will recognize there's no actionable vision and has the judgment to halt or flag it. The existing instruction ("If idea.md is missing or empty, stop immediately") handles the hard failure; the soft failure (present but vacuous) is exactly the kind of judgment call LLMs handle well.

Defining "minimum viable content" in the prompt would require either (a) a word count threshold (brittle, gameable) or (b) a subjective quality bar (which is what the LLM already applies). ## Round 2 — CRITIC

# Round 2 — CRITIC

## (a) Weak Fixes That Won't Actually Work

### The `mktemp` fix for Point 1 still has a failure mode

The defender's accepted fix:
```bash
tmpfile=$(mktemp) && jq '.awaitingApproval = true' {basePath}/.spec-drive-state.json > "$tmpfile" && mv "$tmpfile" {basePath}/.spec-drive-state.json
```

**Problem**: `mktemp` creates a file in `/tmp` by default. The `mv` from `/tmp/tmp.XXXXXX` to `{basePath}/...` may cross filesystem boundaries. On Linux, cross-filesystem `mv` is not atomic — it degrades to copy-then-delete. If the process is killed mid-copy (OOM, signal, timeout), you get a truncated state file.

**Failure scenario**: The spec directory lives on a mounted volume (common in Docker-based dev environments or when `~/spec-drive-projects/` is on a different partition). `mv` silently falls back to `cp + rm`. Process killed during copy → corrupted JSON → every downstream agent breaks.

**Actual fix**: Use `mktemp` *in the same directory as the target file*:
```bash
tmpfile=$(mktemp "{basePath}/.spec-drive-state.json.XXXXXX") && jq '.awaitingApproval = true' "{basePath}/.spec-drive-state.json" > "$tmpfile" && mv "$tmpfile" "{basePath}/.spec-drive-state.json"
```
Same-filesystem `mv` is a single `rename(2)` syscall — actually atomic. This is the standard pattern (how every serious config writer from `git` to `systemd` does it).

---

### The file-existence guard for Point 2 creates a new race

The defender's fix:
```bash
[ -f "{basePath}/.spec-drive-state.json" ] || echo '{}' > "{basePath}/.spec-drive-state.json"
```

**Problem**: This has a TOCTOU (time-of-check-time-of-use) gap. Between the `[ -f ... ]` check and the `echo '{}'`, another process could create the file with valid state. The `echo '{}'` then overwrites it, destroying existing state.

More practically: this guard initializes with `{}` — an empty JSON object. But when `jq '.awaitingApproval = true'` runs against `{}`, it produces `{"awaitingApproval": true}`. That's fine for *this* field. But what about other fields downstream agents expect? If the state file normally contains `{"currentPhase": "research", "startedAt": "...", "awaitingApproval": false}`, initializing with `{}` means all those fields are missing. Downstream agents reading `currentPhase` get `null`. The defender dismissed a "formal schema" but then wrote a guard that needs one to work correctly.

**Failure scenario**: State file was deleted by a filesystem hiccup or user cleanup. Guard creates `{}`. `jq` produces `{"awaitingApproval": true}`. Product-manager agent reads the state file, looks for `currentPhase`, gets `null`, and either crashes or defaults to the wrong phase.

**Actual fix**: The guard should either (a) initialize with a *known-good default document* that includes all expected fields, or (b) refuse to proceed and report that the state file is missing (treating it as an infrastructure error, not something to silently paper over). The defender's "it doesn't need a formal schema" position directly undermines the defender's own fix.

---

### The `codebasePath` default for Point 9 is a trap

The defender accepted adding `codebasePath` but with: *"defaults to CWD if not provided."*

**Problem**: CWD is controlled by the invoking CLI, not by the agent prompt. If the prompt says "defaults to CWD," the agent has no way to *verify* what CWD is or whether it's correct. This "default" turns an explicit gap into an implicit assumption dressed up as a parameter.

**Failure scenario**: A user invokes the research agent from their home directory (`~`). CWD is `~`. The agent globs `**/*.ts` from `~` and picks up files from `node_modules` in an unrelated project, `.nvm` internals, or random scripts in `~/Downloads`. The research.md reports "found 847 TypeScript files with inconsistent patterns" — which is true of the home directory and completely irrelevant to the target project.

The fix doesn't fix the problem unless `codebasePath` is **mandatory with no default**. A bad default is worse than no default because it produces *confidently wrong* output instead of an obvious error. If `codebasePath` isn't provided, the agent should halt and say so — just like it halts when `idea.md` is missing.

### The archive-before-overwrite fix for Point 7 has no cleanup

The defender's fix: *"rename to `research.md.bak-<YYYY-MM-DDTHH:MM>`"*

**Problem**: No retention policy. After 10 re-runs, you have 10 `.bak-*` files in the spec directory. After 50 (across multiple specs over months), the spec directories are cluttered with stale backups that no one reads. The defender rejected `.progress.md` dedup as "complexity without value" — but introduced an unbounded backup accumulation with no cleanup mechanism.

This isn't critical, but it's internally inconsistent: the defender dismissed progress-file rot (Point 8) while introducing a different kind of file rot in the same response. Pick a lane — either file accumulation matters or it doesn't.

**Minimum fix**: Add `"Keep the 3 most recent backups; delete older ones."` One line, no complexity, prevents accumulation.

## (b) NEW Vulnerabilities Introduced by Proposed Changes

### The confidence-signal addition (defender's self-identified gap) has no calibration — the exact problem from Point 6

The defender proposed adding a `Confidence` column with `High/Med/Low` and `[N sources]` annotations. But this has the same problem the defender used to *reject* Point 6: it's subjective labels without a rubric.

When is confidence "High"? Three sources? Five? What if three sources all cite the same original blog post — is that still "High"? What if one source is the official RFC and another is a 2019 Medium post? The defender argued against a risk rubric because "it would make the agent *less* thoughtful" and "risk is inherently contextual." Confidence is equally contextual. By the defender's own logic, a confidence rubric would be counterproductive — the Notes column should carry the reasoning.

The defender can't simultaneously argue:
- ❌ Risk rubric → "makes agent less thoughtful, anchors to predetermined categories"
- ✅ Confidence labels → "gives downstream agents actual signal"

These are the same mechanism (subjective label on a table row). Either structured labels with rubrics add value, or they don't. If they do, the risk rubric critique (Point 6) was valid. If they don't, the confidence proposal is theater.

**To be consistent, either**: (a) accept that both risk and confidence benefit from a lightweight rubric and add one for each, or (b) drop the confidence column and rely on the Notes column for both (which is the defender's own argument for Point 6).

### The web-search failure banner creates a false sense of coverage

The defender accepted: *"mark the External Research section with `⚠️ WEB SEARCH UNAVAILABLE`."*

But this only handles the *total failure* case (tool errors out). The more dangerous case is **partial failure**: WebSearch returns results, but they're shallow, outdated, or from low-quality sources. The agent dutifully records them. No banner appears. The downstream agent sees populated External Research and assumes it's reliable.

**Failure scenario**: WebSearch returns three results for "NextJS server actions best practices." All three are from 2023, before significant API changes in Next.js 14+. The agent records them as findings. No banner. The architect builds on outdated patterns. The code breaks at runtime.

The banner fixes the easy case (tool down) and ignores the hard case (tool returns garbage). The hard case is where the *confidence signal* the defender proposed would actually help — but only if it's implemented with rigor, which brings us back to the rubric problem above.

## (c) Points Where the Defender's Rebuttal Was Insufficient

### Point 3 (malformed `idea.md`): "LLM judgment handles this" is not a design position

The defender's rebuttal: *"It's an LLM. It reads the file and reasons about its content. If the file says 'TBD,' the agent will recognize there's no actionable vision."*

This is hoping, not designing. The defender simultaneously argues:
- The prompt must have `<mandatory>` tags because emphasis markers change LLM behavior (Point 15 defense)
- The prompt must say "NEVER fabricate" because explicit instructions reduce hallucination (Point 13 defense)
- But the prompt *doesn't need* to say "halt if the vision is too vague to research" because the LLM will just... figure it out

If explicit instructions matter (which they do, and the defender argues they do), then this gap needs an explicit instruction. It doesn't need a word count or regex — it needs: *"If the extracted vision is too vague to generate specific search queries, halt and report that idea.md needs more detail."* That's one sentence. It converts an implicit hope into an explicit instruction. The defender's rejection is inconsistent with their own defense of other explicit instructions.

### Point 6 (feasibility calibration): The Notes column defense doesn't survive contact with reality

The defender argued: *"'Requires undocumented third-party API auth' in the Notes column communicates risk better than 'High' with a rubric checkbox."*

This assumes the agent *will* write useful notes. But there's no instruction to do so. The template shows:

```
| [name] | S/M/L/XL | High/Med/Low | [why] |
```

`[why]` is a placeholder, not an instruction. The defender is counting on the agent interpreting a markdown placeholder as "write a substantive risk justification." Compare this to the Quality Commands section, which has an explicit instruction: *"Record what exists and what does NOT exist."* The feasibility section has no equivalent instruction for what belongs in Notes.

**Failure scenario**: The agent fills the Notes column with tautologies: `"Medium complexity"`, `"Standard implementation"`, `"Some integration work"`. These are the LLM equivalent of filler. The Notes column carries zero signal. The downstream architect has a table of labels and vibes.

The fix isn't a rigid rubric — I'm not re-arguing that. The fix is one sentence of instruction: *"In the Notes column, state the specific factor driving the risk rating (e.g., undocumented dependency, novel algorithm, well-trodden CRUD pattern)."* That's guidance, not a rubric. The defender should accept it.

---

### Point 10 (no output validation): "Human approval gate IS the validation" is circular

The defender argued the human review step makes self-validation redundant. But the approval gate's effectiveness depends on the quality of what it's reviewing. A half-empty research.md with the correct template structure *looks* complete on a quick scan. The human gate works when the artifact is clearly good or clearly bad — it fails on artifacts that are *plausibly* complete but subtly hollow.

This isn't about whether the human *should* catch it. It's about whether the prompt can cheaply reduce the failure rate before it reaches the human. One line:

*"Before writing research.md, verify that every section in the template contains substantive content. If any section could not be completed, include `⚠️ INCOMPLETE — [reason]` rather than leaving it empty or filling it with placeholder text."*

This costs nothing, is consistent with the already-accepted web-search banner pattern (Point 4), and reduces the surface area the human has to audit. The defender accepted failure banners for web search but rejected them for the output as a whole — that's inconsistent.
