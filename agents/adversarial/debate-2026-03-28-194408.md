## Round 1 — CRITIC

# Critique: Product Manager Agent Specification

## Framing

**Content type**: Agent behavior specification (prompt definition) for an AI sub-agent that transforms `idea.md` + `research.md` into `requirements.md`.

**"Improved" means**: The spec produces requirements that are internally consistent, complete enough for downstream agents (architect, task planner, executor) to consume without ambiguity, and handles edge cases and failure modes that will absolutely occur in real usage.

---

## Critical Issues (ordered by damage potential)

### 1. No Validation of Input Quality — Garbage In, Garbage Out

The spec says both `idea.md` and `research.md` are "mandatory" and the agent should stop if either is missing. But **existence ≠ quality**. There is zero guidance for what happens when:

- `idea.md` has no "Vision" section or no "Constraints" section (Step 1 explicitly depends on both).
- `research.md` has no "Executive Summary", no "Feasibility Assessment", or no "Open Questions" (Steps 1, 5, and 6 depend on these specific sections).
- Either file is a stub with a single sentence.

**Failure scenario**: Someone creates a minimal `idea.md` with just a title and a paragraph. The agent reads it, finds no "Constraints" section, and either hallucinates constraints or produces NFRs and boundaries with no grounding. Downstream agents build architecture on phantom requirements. Nobody notices until implementation.

**Fix**: Define required sections for each input file explicitly. If a required section is missing or empty, stop and report which section is missing — same as the file-missing check. Add a validation step before Step 1.

---

### 2. The `<mandatory>` Tags Are Meaningless Without Enforcement

The spec uses `<mandatory>` tags around AC format rules and progress updates. This is prompt-level decoration. There is no mechanism to verify compliance. The agent can produce `AC-1.a` instead of `AC-1.1`, skip ACs entirely for a story, or forget to update `.progress.md`.

**Failure scenario**: The agent writes beautiful user stories but formats two acceptance criteria as plain prose instead of `AC-X.Y`. The downstream QA agent (`qa-engineer`) parses for `AC-X.Y` patterns, finds nothing, and either crashes or reports false failures.

**Fix**: Add a self-verification step at the end: "Before writing the file, verify: (a) every US-N has at least one AC-N.Y, (b) every FR-N traces to a US-N, (c) `.progress.md` was updated." Better yet, define a schema or checklist the agent must execute against its own output.

---

### 3. Priority Assignment Is Completely Undefined

Step 4 says priority "reflects impact on the core vision from idea.md" but provides no rubric. What makes something "High" vs "Medium"? Is it based on user impact? Business value? Technical risk? Dependency chain position?

**Failure scenario**: The agent marks 80% of requirements as "High" because everything seems important when you're reading a vision document. The architect and task planner receive a flat priority list with no meaningful differentiation. Sprint planning becomes guesswork.

**Fix**: Define a 2x2 or decision tree. Example: High = blocks core user outcome AND mentioned in vision. Medium = supports core outcome OR addresses a constraint. Low = nice-to-have, not tied to success criteria. Cap High at ~30% of total FRs.

---

### 4. No Traceability Enforcement Between Sections

Step 4 says "Every FR must trace back to at least one user story" but there's no format for encoding that trace. The FR table has columns `ID | Requirement | Priority | Verification` — **there is no column for the source user story**.

**Failure scenario**: The agent writes 12 FRs with no indication which US they came from. When a user story is descoped, nobody knows which FRs to drop. The architect builds components for orphaned requirements.

**Fix**: Add a `Source` column to the FR table: `| ID | Requirement | Source | Priority | Verification |`. Same for NFRs — trace them to constraints in `idea.md` or findings in `research.md`.

---

### 5. Success Criteria Are Disconnected From Acceptance Criteria

Step 6 says success criteria should "Reference specific FR/NFR IDs." But there's no mechanism to ensure these references are valid. More fundamentally, there's no guidance on the **relationship** between success criteria and ACs. Are success criteria aggregations of ACs? Independent measures? Post-launch metrics?

**Failure scenario**: The agent writes success criteria like "SC-1: All FR-1 through FR-5 are met" — which is just a restatement of "we shipped the requirements." This tells nobody whether the project actually achieved its goal. Or it writes "SC-1: User adoption increases 20%" which is unmeasurable by the executor agents.

**Fix**: Constrain success criteria to be measurable by the system at development/test time (not post-launch business metrics). Require each to specify an evaluation method: "SC-1: A new user can complete onboarding without documentation, verified by integration test X."

---

### 6. The 3000-Word Limit Will Silently Truncate Complex Projects

The constraint says "Keep the document under 3000 words." For a project with 8+ user stories, each with 3-5 ACs, plus FR/NFR tables, boundaries, dependencies, glossary — 3000 words is tight. The agent will either:

(a) Drop user stories or ACs to fit the budget, or
(b) Write terse, ambiguous requirements that save words but lose meaning.

**Failure scenario**: A complex project gets a requirements doc where the last 3 user stories have 1 AC each instead of 3-4, because the agent hit the word limit. The architect designs those features with incomplete requirements. Rework during implementation.

**Fix**: Either remove the hard limit and replace it with "be concise — one sentence per requirement, no explanatory prose," or add guidance: "If the project requires more than 8 user stories, split into sub-specs and reference them." Give the agent a release valve.

---

### 7. No Conflict Resolution Between idea.md and research.md

Step 1 says to read both files. But what happens when they contradict each other? `idea.md` might say "must support offline mode" while `research.md` concludes "offline support is infeasible with the chosen stack."

**Failure scenario**: The agent includes "offline support" as FR-3 (because idea.md said so) while simultaneously noting in Dependencies that the required library doesn't support offline. The architect receives contradictory requirements and has to guess which source to trust.

**Fix**: Add an explicit conflict resolution rule: "If research.md's feasibility assessment contradicts a goal in idea.md, flag the conflict in a 'Risks & Conflicts' section. Do NOT include the conflicting item as a confirmed requirement. Reference both sources."

---

### 8. `.progress.md` Update Is Append-Only With No Structure

The spec says "Append only. Never delete existing learnings." But it doesn't define the format of `.progress.md`, what "Current Task" looks like, or how learnings are structured. It also doesn't handle the case where `.progress.md` doesn't exist yet.

**Failure scenario**: `.progress.md` doesn't exist. The agent either crashes or creates it with an arbitrary format. The next agent in the pipeline expects a different format and can't parse it. Or: after 15 spec iterations, `.progress.md` is 200 lines of unstructured learnings that nobody reads.

**Fix**: Define the `.progress.md` schema explicitly (or reference where it's defined). Specify create-if-missing behavior. Cap learnings to top-N most relevant or require tagging by phase.

---

### 9. "Role" Selection in User Stories Is Unguided

Step 2 says "Choose the role that best represents who benefits." The examples given are generic (developer, end user, operator). There's no instruction to derive roles from `idea.md` or `research.md`, and no constraint against inventing roles not mentioned in the inputs.

**Failure scenario**: The agent invents "system administrator" as a role for 3 user stories, but the project is a CLI tool with no admin concept. Downstream agents design admin interfaces that nobody asked for.

**Fix**: Add: "Roles must be derived from the personas, users, or stakeholders mentioned in idea.md. If a role is implied but not explicit, name it and add it to the Glossary."

---

### 10. No Handling of Partial Output / Interruption

If the agent is interrupted mid-generation (timeout, token limit, crash), `.progress.md` won't be updated and `requirements.md` may be a partial file. There's no idempotency or resume mechanism.

**Failure scenario**: The agent writes 70% of `requirements.md` and times out. The file exists but is incomplete. The next pipeline stage (architect) reads a truncated requirements file and designs against partial requirements. Nobody notices the truncation because the file exists.

**Fix**: Write to a temp file first (`requirements.md.tmp`), then atomically rename on completion. Or add a frontmatter field `status: complete|draft` that the agent sets to `complete` only at the very end. Downstream agents check this field.

---

## Round 1 — DEFENDER

# Defense: Product Manager Agent Specification

## The Critic's Blind Spot — What Was Missed

Before addressing the ten points: **the critic focused almost entirely on structural and mechanical concerns while missing the most consequential gap in this spec.**

**The agent has no protocol for surfacing ambiguity.** The spec tells the agent to stop if files are missing. It tells the agent to note unresolved open questions as dependencies. But it says *nothing* about what to do when `idea.md` is genuinely ambiguous — when two reasonable readings of the vision lead to fundamentally different requirement sets. The agent is forced to silently pick an interpretation.

This is more damaging than any of the critic's ten points because:

1. **It's invisible downstream.** A missing traceability column (Point 4) or a flat priority list (Point 3) are problems the architect can *see* and flag. A silently-chosen interpretation of an ambiguous vision looks like a correct, confident requirement. Nobody questions it until implementation reveals the mismatch.
2. **It's the most likely failure.** Visions are *frequently* ambiguous. "Make the tool easy to configure" could mean "GUI wizard" or "well-documented YAML." The agent will pick one and write user stories for it. The whole pipeline marches forward on a coin flip.
3. **The fix is cheap.** Add a "Clarifications Needed" section that lists ambiguities the agent identified but couldn't resolve from the inputs. Downstream, this becomes a blocker for the architect — or a signal to go back to the human.

The critic's analysis is mechanically competent but missed the *epistemological* problem at the heart of requirements generation. Now, to the specific points.

## Point-by-Point Response

### Point 1: No Validation of Input Quality — PARTIAL CONCEDE

The critic is right that existence ≠ quality. But the severity is overstated.

**What the critic gets wrong:** This is an LLM agent, not a JSON parser. If `idea.md` has no "Constraints" section, the agent doesn't segfault — it reads the whole document and extracts constraints from wherever they appear. Step 1 says "Read idea.md's Vision and Constraints sections" as *guidance for where to look*, not as a rigid section-name parser. LLMs handle structural variation in input documents competently.

**What the critic gets right:** A one-sentence stub `idea.md` is a real problem. The agent shouldn't try to generate 12 FRs from "Build a cool dashboard."

**My fix (better than the critic's):** Don't validate section names — that's brittle and couples this spec to the exact format of `idea.md`. Instead, add a **sufficiency check**: "Before proceeding to Step 1, assess whether the inputs contain enough information to produce at least 2 user stories with testable acceptance criteria. If not, stop and report what's insufficient." This is robust to format variation while catching genuine stubs.

---

### Point 2: `<mandatory>` Tags Without Enforcement — REJECT

The critic frames `<mandatory>` tags as "prompt-level decoration" that is "meaningless without enforcement." This fundamentally misunderstands how LLM agent prompts work.

`<mandatory>` tags are **attention signals** — they tell the model "this matters more than the surrounding text." Research on prompt engineering consistently shows that emphasis markers (XML tags, capitalization, repetition) measurably increase compliance. They're not "meaningless" — they're the *primary* enforcement mechanism available in a prompt specification.

The failure scenario — "the agent writes AC-1.a instead of AC-1.1" — is speculative. The spec gives an explicit format example with `AC-1.1`. LLMs are extremely good at following format examples. The critic provides no evidence this failure is *likely*, only that it's *possible*.

**That said,** a self-verification checklist is a genuinely good practice that costs nothing. I'd adopt it not because the `<mandatory>` tags are broken, but because defense in depth is sound engineering:

> **Step 8 (new): Self-Check**
> Before writing the final file, verify: (a) every US-N has at least one AC-N.Y in the correct format, (b) every FR traces to a US, (c) NFR targets are numeric/measurable.

I'm adopting the fix on its merits, not conceding the diagnosis.

---

### Point 3: Undefined Priority Rubric — CONCEDE

The critic is right. "Reflects impact on the core vision" is too vague to produce consistent prioritization across runs. The failure scenario (80% marked High) is not hypothetical — it's the *default behavior* of LLMs when given vague prioritization guidance. They err toward "important."

**However, I reject the critic's specific fix.** A hard cap of "~30% High" is an arbitrary number that will force the agent to deprioritize something critical in a project that genuinely has many critical requirements. The cap creates a different failure: the agent plays Tetris with priorities to hit a percentage target instead of reasoning about impact.

**Better fix — a decision rubric without artificial caps:**

> - **High**: Directly enables a Success Criterion. Without this FR, the project cannot meet its stated goals.
> - **Medium**: Supports a Success Criterion or addresses a constraint from idea.md. Project succeeds without it, but with gaps.
> - **Low**: Enhances quality or usability. Not tied to any Success Criterion.
>
> If more than 50% of FRs are High, re-examine whether the scope is too broad for a single spec.

This ties priority to the spec's *own* success criteria (creating an internal feedback loop) and uses the 50% threshold as a diagnostic signal rather than an enforcement cap.

---

### Point 4: Missing Traceability Column — CONCEDE

Clean hit. The spec says "Every FR must trace back to at least one user story" but the table format has no column for it. That's a genuine oversight — the rule exists but the structure doesn't support it.

I accept the critic's fix: add a `Source` column to the FR table. Simple, zero-cost, immediately useful.

```markdown
| ID | Requirement | Source | Priority | Verification |
|----|-------------|--------|----------|--------------|
| FR-1 | [Description] | US-2 | High | [How to verify] |
```

I'd extend it to the NFR table as well, tracing to either a constraint in `idea.md` or a finding in `research.md`:

```markdown
| ID | Requirement | Source | Metric | Target |
|----|-------------|--------|--------|--------|
| NFR-1 | [Description] | idea.md/Constraints | [Measure] | [Threshold] |
```

### Point 5: Success Criteria Disconnected — PARTIAL CONCEDE

The critic identifies a real tension but overstates it.

**What the critic gets wrong:** The spec already says success criteria should "Reference specific FR/NFR IDs." That *is* a connection mechanism. And the example "SC-1: All FR-1 through FR-5 are met" isn't a plausible output — it's a strawman. An LLM following "Define 2-4 measurable outcomes that determine if the project succeeded" and "higher-level than ACs" will produce something more meaningful than "we shipped the requirements."

**What the critic gets right:** The guidance is ambiguous about what *kind* of measurement is appropriate. "User adoption increases 20%" is indeed unmeasurable by executor agents.

**Fix — constrain the measurement scope:**

> Success criteria must be verifiable at development/test time by the agents in this pipeline. They answer "did we build what we intended?" not "did the market respond?" Each must specify a verification method (test, demonstration, inspection).

This is tighter than the current spec without being as restrictive as the critic's proposal (which would eliminate legitimate success criteria like "The system handles 1000 concurrent users" — measurable at test time via load testing but not via "integration test X").

---

### Point 6: 3000-Word Limit Will Silently Truncate — REJECT

The critic describes a *possible* problem but provides no evidence it's *likely* given the actual usage context.

This agent operates within the Ralph Specum pipeline, which is designed for **focused, single-feature specs** (note the `basePath` pattern: `P042-feature`). These are not enterprise-scale PRDs — they're scoped feature specifications. A well-written requirements doc for a focused feature with 4-6 user stories, 8-12 FRs, and 4-6 NFRs fits comfortably in 1500-2000 words.

The 3000-word limit is a **guardrail against bloat**, not a straitjacket. The spec's own principle (from the broader project context) is "Simple is smart" and "concise requirements are usable requirements." Removing the limit invites the opposite failure: a 6000-word requirements doc that nobody reads, with redundant prose and over-specified details that belong in the design phase.

**If** a project genuinely needs 8+ user stories with 4+ ACs each, the right answer isn't a longer requirements doc — it's splitting the spec. The pipeline already supports multiple specs. The limit is a design signal: "If you can't fit it, you're scoping too broadly."

I'd add one sentence of guidance rather than removing the limit:

> If requirements approach the 3000-word limit, consider whether the scope should be split into multiple specs.

---

### Point 7: No Conflict Resolution — CONCEDE

The critic is right that input conflicts will occur and the spec doesn't address them. The example is concrete and plausible: vision says "offline mode," research says "infeasible." This will happen.

I accept the concept but **propose a leaner fix** than a full "Risks & Conflicts" section (which adds structural overhead to every requirements doc, even conflict-free ones):

> **Conflict resolution rule**: If research.md contradicts or downgrades the feasibility of a goal in idea.md, do NOT include the conflicting item as a confirmed FR. Instead, add it to Out of Scope with the annotation: "Descoped: [brief reason from research.md]. See research.md [section] for details." If the conflict is ambiguous (research raises concerns but doesn't rule it out), include the FR but add a dependency note: "Depends on resolution of [conflict description]."

This reuses the existing Out of Scope and Dependencies sections rather than creating a new section. It handles both clear conflicts and ambiguous ones. And it doesn't add noise to clean specs.

---

### Point 8: `.progress.md` Format Undefined — REJECT

The critic assumes this spec should define the `.progress.md` schema. It shouldn't. `.progress.md` is a **pipeline-level artifact** shared across all agents (product manager, architect, task planner, executor, QA). Its format is defined by the pipeline orchestrator, not by each individual agent spec.

This agent's responsibility is limited to: set Current Task, append learnings. It's a *consumer* of the format, not its *owner*. Defining the schema here would create coupling — if the schema changes, every agent spec needs updating.

The "doesn't exist yet" scenario is also a pipeline concern. The orchestrator creates `.progress.md` when the spec is initialized (when `idea.md` is written). If it doesn't exist when this agent runs, something went wrong upstream — and this agent should report the error, just like it does for missing input files.

**Minimal addition that addresses the valid concern without overstepping:**

> If `.progress.md` does not exist, create it with the fields `Current Task` and `Learnings`. If it exists, preserve its existing structure and append only.

---

### Point 9: Unguided Role Selection — REJECT

The failure scenario is a stretch. The critic claims the agent might "invent 'system administrator' as a role for 3 user stories" for a CLI tool. This requires the agent to:

1. Read an `idea.md` that describes a CLI tool
2. Read a `research.md` that analyzes a CLI tool
3. Somehow conclude that a "system administrator" is a relevant user

LLMs don't do this. They're *extremely* good at inferring user roles from context. "Choose the role that best represents who benefits" combined with the actual content of the input documents is sufficient guidance. The agent derives roles from the inputs *naturally* because that's all the context it has.

The proposed fix ("Roles must be derived from the personas, users, or stakeholders mentioned in idea.md") adds a rigid constraint that could *hurt*: if `idea.md` mentions only "users" generically, the agent can't differentiate between "developer using the API" and "operator monitoring the system" — both legitimate roles that might not be explicitly named in a vision document.

Adding discovered roles to the Glossary is a fine idea, though. I'd add:

> If you introduce a role not explicitly named in idea.md, add it to the Glossary with a definition.

---

### Point 10: No Partial Output Safety — REJECT

This is an infrastructure concern masquerading as a spec concern.

Atomic file writes, temp files, and crash recovery are the responsibility of the **execution environment** (the Ralph Specum pipeline, the agent SDK, the filesystem layer), not of a prompt specification. The product manager agent prompt shouldn't contain instructions about `requirements.md.tmp` and `mv` operations — that's implementation detail that violates the spec's own principle of describing WHAT, not HOW.

The `status: complete|draft` frontmatter idea has some merit but introduces a new failure mode: the downstream agent now has to check a field that might be missing (in older specs), might be wrong (agent sets it to `complete` before actually finishing due to a prompt bug), or might create a deadlock (architect waits for `complete`, agent crashed before setting it).

The real safety mechanism already exists: the `.progress.md` update happens **after** writing `requirements.md`. If the agent crashes mid-write, `.progress.md` still says the previous task, and the orchestrator knows to re-run. This is the standard pipeline pattern — the progress marker IS the completion signal.

## Strengths the Critic Missed

The critic analyzed only weaknesses. For balance, this spec has several design strengths worth preserving:

1. **The AC-X.Y numbering scheme** creates a machine-parseable, cross-referenceable identifier system. Downstream agents can reference `AC-3.2` unambiguously. This is better than most requirements specs I've seen.

2. **The separation of FR/NFR tables from user stories** is a deliberate design choice. User stories capture *intent*; FR tables capture *commitments*. Having both lets the architect work from the FR table (structured, prioritized) while the QA agent validates against ACs (testable, binary). This dual structure serves the pipeline well.

3. **The "Do NOT include implementation details" constraint** is critical and underappreciated. Requirements docs that leak implementation choices are the #1 cause of over-constrained designs. This spec correctly draws the line.

4. **The Glossary section** prevents a subtle but common pipeline failure: term drift across agents. If the PM defines "workspace" and the architect interprets it differently, everything downstream is wrong. The glossary is a shared vocabulary contract.

## Round 2 — CRITIC

# Round 2 — CRITIC

## Focus: The Defender's Fixes Are Weaker Than They Look

The defender conceded four points and adopted fixes for two more. Good. But several of those fixes are cosmetic — they name the problem without solving it. And the defender's own contribution (ambiguity surfacing) introduces a new structural problem. Let me go through what will actually break.

---

### A. The "Sufficiency Check" (Point 1 Fix) Is Unverifiable by Design

The defender rejected section-name validation as "brittle" and proposed instead:

> "Assess whether the inputs contain enough information to produce at least 2 user stories with testable acceptance criteria. If not, stop and report what's insufficient."

This sounds reasonable. It is not. Here's why it will fail:

**The sufficiency threshold is defined in terms of the agent's own output capability.** You're asking an LLM: "Can you produce 2 user stories from this?" The answer is *always yes*. An LLM can generate 2 user stories from a napkin sketch. It can generate 10 user stories from the sentence "Build a dashboard." The stories will be hallucinated garbage, but the sufficiency check passes because the check measures *generative capacity* (unlimited) rather than *input signal quality* (finite).

This is the equivalent of asking "Do you have enough ingredients to cook dinner?" to someone who can conjure ingredients from thin air. The answer is structurally guaranteed to be yes.

**What actually works:** A check grounded in *input properties*, not *output potential*. Example: "The inputs must collectively describe at least: (a) who the users are, (b) what problem is being solved, (c) at least one concrete constraint or boundary. If any of these three are absent, stop." This is checkable against the input text without requiring the agent to predict its own output quality — which it cannot do reliably.

---

### B. The Priority Rubric Creates a Circular Dependency

The defender proposed:

> - **High**: Directly enables a Success Criterion.
> - **Medium**: Supports a Success Criterion or addresses a constraint.
> - **Low**: Not tied to any Success Criterion.

The problem: **Success Criteria are defined in Step 6. FR priorities are assigned in Step 4.** The agent hasn't written the Success Criteria yet when it's assigning priorities.

The defender might argue the agent can "think ahead" to what success criteria will be. But that means priorities are based on *imagined future output*, not on any artifact that exists at processing time. In practice, the agent will either:

1. Assign priorities in Step 4 based on vibes (the current broken behavior), then retroactively ensure the Success Criteria in Step 6 happen to align — making the rubric a post-hoc rationalization, not a decision tool.
2. Skip ahead to draft Success Criteria mentally, then come back — which means the linear step order in the spec is a lie, and the agent is doing two passes whether the spec says so or not.

**Fix:** Either reorder the steps (define Success Criteria *before* the FR table), or decouple the priority rubric from Success Criteria entirely. A rubric based on the *input documents* works without forward-references: "High = directly stated as a goal in idea.md's vision. Medium = implied by a constraint or finding in research.md. Low = inferred by the agent but not explicitly grounded in either input."

---

### C. The Conflict Resolution Fix Has a Scoping Hole

The defender's conflict resolution rule:

> "If research.md contradicts a goal in idea.md, do NOT include it as a confirmed FR. Instead, add it to Out of Scope with annotation."

This handles **binary conflicts** (feasible vs. infeasible). It does not handle **degree conflicts**, which are far more common:

- `idea.md`: "Support all major browsers"
- `research.md`: "Safari has partial support for the required API; workaround exists but adds 2 weeks"

Is this a contradiction? Research didn't say "infeasible." It said "feasible but costly." The defender's rule gives no guidance here. The agent includes it as an FR with no annotation, the architect plans for it, and the task planner discovers the 2-week hit during implementation.

The second clause ("if ambiguous, include the FR but add a dependency note") partially addresses this, but the trigger condition — "research raises concerns but doesn't rule it out" — requires the agent to judge the severity of a concern. That's the same subjective judgment gap the defender criticized in the critic's priority cap.

**Fix:** Replace the binary trigger with a graduated one: "If research.md identifies feasibility risks, cost implications, or partial support for a goal in idea.md, include the FR but add a `Risk` annotation in the Source column: `US-3 [Risk: partial Safari support, +2wk est.]`. This surfaces the information without requiring the agent to make a feasibility judgment call."

---

### D. The "Clarifications Needed" Section (Defender's New Contribution) Creates a Pipeline Deadlock

The defender's most significant contribution was identifying the ambiguity problem and proposing:

> Add a "Clarifications Needed" section that lists ambiguities. Downstream, this becomes a blocker for the architect — or a signal to go back to the human.

**"Or" is doing all the heavy lifting in that sentence.** What *actually* happens when the architect reads a Clarifications Needed section with 3 items?

Option 1: The architect treats them as blockers and stops. Now the pipeline is halted, waiting for human input. In the Ralph Specum pipeline, this means the automated execution loop (`/implement`) stalls. The human might not check for hours. This is a deadlock by design.

Option 2: The architect ignores the clarifications and proceeds with assumptions. Now the section is noise — it exists, nobody acted on it, and the same silent-interpretation problem occurs, just one stage later.

Option 3: The pipeline orchestrator intercepts the clarifications and routes them to the human before invoking the architect. This requires orchestrator-level logic that doesn't exist (the defender acknowledged `.progress.md` is a pipeline-level concern — so is this).

The defender identified a real problem but proposed a solution that requires pipeline infrastructure changes while simultaneously arguing (in Points 8 and 10) that pipeline-level concerns are out of scope for this spec. **You can't have it both ways.** Either this spec can impose pipeline-level requirements, in which case Points 8 and 10 are back on the table, or it can't, in which case the Clarifications section is aspirational fiction.

**Fix:** Make the section self-contained within this spec's authority. Instead of creating a downstream blocker, require the agent to *resolve* ambiguities using the inputs it has, then *document* the interpretation chosen:

> **Assumptions & Interpretations**: For each ambiguity in the inputs, state the interpretation chosen and the reasoning. Example: "idea.md says 'easy to configure' — interpreted as 'YAML-based configuration with schema validation' based on research.md's finding that target users are developers comfortable with config files."

This doesn't block the pipeline, doesn't require orchestrator changes, and — critically — makes the silent interpretation *visible*. The human or architect can review the assumptions and flag disagreements. It solves the defender's stated problem (invisible interpretation choices) without creating a new one (pipeline deadlock).

### E. The Self-Verification Step (Point 2 Fix) Checks Structure, Not Semantics

The defender adopted a self-verification step:

> "Before writing the final file, verify: (a) every US-N has at least one AC-N.Y in the correct format, (b) every FR traces to a US, (c) NFR targets are numeric/measurable."

This catches *formatting* errors. It does not catch the errors that actually matter:

- **AC that restates the user story.** US-3: "As a developer, I want to configure the tool via YAML." AC-3.1: "The tool can be configured via YAML." This passes the self-check (correct format, exists, traces to US-3) while adding zero testable information.
- **FR with no meaningful verification.** FR-5: "System supports error handling." Verification: "Unit test." This passes the check (has a verification column entry) while telling the QA agent nothing about what to test.
- **NFR with a technically-numeric but meaningless target.** NFR-2: "Code maintainability." Metric: "Cyclomatic complexity." Target: "< 1000." This passes check (c) — the target is numeric! — while being functionally useless.

The self-check creates a **false sense of rigor**. It's a lint pass that catches syntax errors while the logic bugs sail through.

**Fix:** Add at least one semantic check: "Verify that no AC merely restates its parent user story — each AC must specify a *condition* not already stated in the story's 'I want to' clause." And for FRs: "Verify that each Verification entry names a specific test type AND what it validates (not just 'unit test' but 'unit test verifying [specific behavior]')."

### F. The Defender's Rejection of Point 6 (Word Limit) Relies on an Assumption About Project Scope

The defender argued the 3000-word limit is fine because the pipeline handles "focused, single-feature specs" and a well-written one fits in 1500-2000 words.

This argument rests entirely on the claim that specs are always narrowly scoped. But the spec itself imposes **no scope constraint on `idea.md`**. There is no rule saying "idea.md must describe a single feature." If someone feeds in a broad `idea.md` (which will happen — humans don't naturally scope to single features), the product manager agent faces an impossible choice: honor the input scope and bust the word limit, or silently narrow the scope to fit.

The defender proposed adding: "If requirements approach the 3000-word limit, consider whether the scope should be split into multiple specs."

**"Consider" is not actionable.** The agent cannot split specs — it writes one file. "Consider" means "notice the problem and do nothing about it." The agent will generate the full requirements, hit the limit, and either truncate or ignore the guidance. There is no mechanism for the agent to *actually split* — it would need to create multiple `idea.md` files, invoke itself recursively, or signal the orchestrator. None of these are available.

**Fix:** Make the guidance operational: "If the inputs would produce more than 8 user stories, stop and report: 'Scope exceeds single-spec capacity. Recommend splitting into N specs along these boundaries: [list feature groupings].' Do not attempt to produce a truncated requirements doc." This gives the agent a concrete action (stop + report) rather than an unenforceable suggestion ("consider").

### G. The Defender Silently Dropped Point 9's Core Problem

The defender rejected Point 9 (unguided role selection) and proposed: "If you introduce a role not explicitly named in idea.md, add it to the Glossary."

This mitigates *term confusion* but doesn't address the *structural* problem: user stories keyed to the wrong role produce wrong requirements. If the agent picks "API consumer" when the actual user is "CLI operator," the acceptance criteria will test the wrong interaction model. Adding "API consumer" to the Glossary with a clean definition doesn't fix the fact that the stories are written for the wrong person.

The defender argued LLMs "naturally" infer roles from context. This is true *when the context is unambiguous.* When `idea.md` says "users can configure the system" without specifying whether "users" are developers, ops engineers, or end-users — and these three groups have fundamentally different configuration needs — the LLM picks one silently.

This connects directly to the ambiguity problem the defender raised. If the "Assumptions & Interpretations" section I proposed in (D) is adopted, role selection becomes a specific instance: "idea.md says 'users' — interpreted as 'DevOps engineers' based on research.md's analysis of the target deployment environment." That's the real fix. The Glossary addition alone is cosmetic.

## Responding to the Meta-Argument

The critic closes with: "An agent prompt that says 'consider whether X' is functionally equivalent to one that says nothing."

This is empirically false. Prompt engineering research (and practical experience running LLM agent pipelines) consistently shows that guidance-level instructions ("consider," "prefer," "aim for") produce measurably different output distributions than no instruction at all. They don't produce *guaranteed* compliance — but neither do "stop and report" instructions, which the agent can also fail to follow.

The real spectrum isn't binary (hard instruction vs. nothing). It's:

1. **Format rules** (AC-X.Y numbering) — highest compliance, easy to verify
2. **Conditional stops** (missing file → stop) — high compliance, binary trigger
3. **Decision rubrics** (priority = High if enables Success Criterion) — moderate compliance, requires judgment
4. **Guidance** ("consider splitting") — lowest compliance, but non-zero

The spec should use the strongest level appropriate for each concern. I've adopted harder formulations where the critic showed mine were too soft (Points A, B, D). But I won't pretend that every instruction can or should be a conditional stop.
