## Round 1 — CRITIC

# Critique: `qa-engineer` Agent Specification

**Content type:** Agent system prompt / behavioral specification for a QA verification subagent in a spec-driven development pipeline.

**"Improved" means:** The agent reliably catches real failures, never produces false passes, handles its own edge cases gracefully, and produces output that actually enables one-pass fixes downstream.

---

## 1. Critical Gap: No Handling of Missing or Malformed Inputs

The spec says the agent receives a `[VERIFY]` task block, a `basePath`, and `.progress.md` content. But it never defines what happens when:

- **`basePath` doesn't exist or is inaccessible.** The agent will attempt to read `{basePath}/requirements.md` and write to `{basePath}/.progress.md`. If the path is wrong, the agent silently fails or hallucinates.
- **The `[VERIFY]` block has no `Verify command`.** Step 1 says "Execute each Verify command from the task block" — but what if there are none? The agent has no fallback. It could vacuously pass (every command in the empty set succeeded), which is a **false positive**.
- **`requirements.md` doesn't exist.** Step 2 says "Read the AC definition from requirements.md." If the file is missing, the agent either crashes or skips AC checks entirely — both catastrophic.
- **`.progress.md` doesn't exist yet.** The Progress Update section says "Append only. Never delete existing learnings." But the first verification task in a project has no `.progress.md` to append to.

**Failure scenario:** First task in a new spec. `basePath` exists but `.progress.md` and `requirements.md` haven't been created yet. Agent tries to read both, gets errors, and either crashes or silently skips AC validation and progress logging. The executor gets no retry context. The next run repeats the same failure.

**Fix:** Add an explicit "Input Validation" step before Step 1. Define behavior for each missing input: create `.progress.md` if absent, FAIL explicitly if `requirements.md` is missing and ACs are referenced, FAIL if no Verify commands exist (don't vacuously pass).

---

## 2. The Mock Anti-Pattern Scan (Step 3) Is Unscoped and Unreliable

The step says "Scan test files **touched by the preceding tasks**." But the agent receives only the `[VERIFY]` task block, `basePath`, and `.progress.md`. There is no mechanism to determine which files were "touched by the preceding tasks."

- The agent doesn't receive a file diff or changeset.
- `.progress.md` may mention files but there's no guaranteed structured format for that.
- The agent would need to run `git diff` against some baseline — but no baseline commit is specified.

**Failure scenario:** The agent can't determine which test files to scan. It either scans nothing (silently skipping the mandatory step), scans everything (wasting time, producing noise), or guesses wrong (missing the actual problematic files while flagging irrelevant ones).

Additionally, the thresholds are vague:

- **"More than 70% of test lines are mock setup vs actual assertions"** — How does an LLM reliably count this? What counts as "mock setup" vs "assertion"? Is `jest.fn()` a mock line? Is `expect(mockFn).toHaveBeenCalled()` an assertion or a mock assertion? This is asking for subjective judgment dressed up as a quantitative check.
- **"No integration path"** — the agent scans individual files. It cannot determine whether integration tests exist *elsewhere in the codebase* without a full project scan.

**Fix:** Either provide the agent with an explicit file list (from the executor's git diff) or specify a concrete command to derive it (e.g., `git diff --name-only HEAD~N -- '*.test.*'`). Replace the 70% heuristic with a concrete, mechanically-checkable rule, or downgrade it from mandatory to advisory.

---

## 3. "Exit Code 0 Is Required" Contradicts "Unless the Task Expects Failure Output"

The Constraints section states:

> NEVER pass a verification that has failing commands. Exit code 0 is required.

But Step 1 says:

> A non-zero exit code is an immediate FAIL **unless the task expects failure output**.

These contradict. The Constraints section is absolute; Step 1 carves an exception. An LLM agent will resolve this ambiguity inconsistently — sometimes respecting the exception, sometimes following the absolute rule.

**Failure scenario:** A `[VERIFY]` task checks that a CLI exits with code 1 on invalid input (a negative test). The agent runs the command, gets exit code 1, and FAILs because the constraint says "Exit code 0 is required." The executor wastes a retry cycle fixing nothing.

**Fix:** Reconcile the two. The Constraints section should say: "NEVER pass a verification that has unexpected exit codes. Exit code 0 is required unless the task explicitly expects a non-zero exit code, in which case the expected code must match exactly."

---

## 4. Before/After Comparison (Step 4) Has No Baseline

The step says:

> Check git diff or file contents for the expected change.

But no baseline is defined. `git diff` against what? HEAD? HEAD~1? A specific commit? The agent doesn't receive a "before" commit hash.

**Failure scenario:** The executor made the change three commits ago, then made formatting fixes. `git diff HEAD~1` shows only the formatting. The agent concludes the expected change is "missing" and FAILs. Or `git diff` shows unstaged changes from something unrelated and the agent gets confused.

**Fix:** Either require the task block to include a baseline commit reference, or specify a default (e.g., diff against the commit hash recorded in `.progress.md` when the preceding task started).

---

## 5. Writing to `.progress.md` Creates a Race Condition

Both the "Failure Context for Retries" and "Progress Update" sections mandate writing to `.progress.md`. In the Ralph Specum workflow, the `spec-executor` agent also writes to `.progress.md`. If verification fails and the executor immediately retries, both agents could write concurrently.

Even without concurrency, the spec doesn't define the *format* of existing `.progress.md` entries. The agent is told to "Append only" and to update "Current Task" — but updating "Current Task" is a *mutation*, not an append. These are contradictory instructions.

**Failure scenario:** `.progress.md` has `Current Task: Implement auth module`. The QA agent is told to "Set Current Task to the failed task description" (a replacement) AND to "Append only. Never delete existing learnings" (no replacements). The agent either duplicates the Current Task line or corrupts the file structure, breaking parsing for the next agent.

**Fix:** Define the exact `.progress.md` schema. Separate the "append learnings" operation from the "update current task" operation. Acknowledge that "update current task" is a replace, not an append, and provide the exact edit pattern (e.g., "Replace the line starting with `Current Task:` with the new value").

---

## 6. No Timeout or Resource Bounds on Verify Commands

Step 1 says "Execute each Verify command from the task block" — with zero guardrails. The verify command could be:

- `npm test` on a suite that takes 20 minutes
- A command that hangs indefinitely (e.g., starts a server and never exits)
- A command that produces 500MB of stdout
- A destructive command (`rm -rf /`, `git push --force`)

**Failure scenario:** A verify command is `docker-compose up && curl localhost:3000/health`. The `docker-compose up` never exits. The agent hangs until the parent process's watchdog kills it (if one exists). No verification output is produced. No `.progress.md` update happens. The system is stuck.

**Fix:** Mandate a timeout for each command (e.g., 120 seconds). Require the agent to validate commands against a deny-list of destructive patterns before execution. Specify max stdout capture size.

---

## 7. The AC Check (Step 2) Has No Defined Verification Method

Step 2 says:

> Check the actual implementation against the AC

But *how*? The agent has already run the Verify commands in Step 1. If the AC says "Users can reset their password via email," how does the agent check this beyond what the Verify commands already tested? It would need to:

- Read source code (not specified)
- Run additional commands (not specified)  
- Make judgment calls about whether code "satisfies" a natural-language AC (subjective, unreliable)

This step is either redundant with Step 1 (if the Verify commands already cover the ACs) or impossible to execute mechanically (if they don't).

**Failure scenario:** The Verify command is `npm test -- --coverage`. It passes. AC-1.3 says "Error messages must be user-friendly." The agent has no way to check this from test output alone. It either rubber-stamps it (false pass) or flags it speculatively (false fail with no evidence).

**Fix:** Either require every AC to have a corresponding Verify command (making Step 2 a cross-reference check, not a separate verification), or specify exactly what tooling the agent should use for AC validation (code grep patterns, additional commands, etc.).

---

## 8. "Exactly One Signal" Is Fragile in Practice

The spec demands the output ends with exactly `VERIFICATION_PASS` or `VERIFICATION_FAIL`. But:

- LLMs sometimes add trailing whitespace, newlines, or markdown formatting
- The agent might include both tokens in explanatory text ("We did not see VERIFICATION_FAIL conditions, so: VERIFICATION_PASS")
- If the consumer parses with a simple regex like `/VERIFICATION_(PASS|FAIL)/`, the explanatory mention causes ambiguity

**Failure scenario:** The agent writes: "Unlike the previous run which ended in VERIFICATION_FAIL, all issues are resolved. VERIFICATION_PASS." The consumer regex matches both signals. The pipeline breaks or picks the wrong one.

**Fix:** Specify that VERIFICATION_PASS and VERIFICATION_FAIL must *only* appear as the terminal signal. Add an explicit constraint: "Do NOT use the words VERIFICATION_PASS or VERIFICATION_FAIL anywhere except as the final signal line." Better yet, wrap the signal in a structured format: `<!-- RESULT: PASS -->` or a JSON block.

---

## 9. No Idempotency Guarantee

The agent appends to `.progress.md` on every run. If verification is retried 5 times, you get 5 duplicate failure entries. The "Append only. Never delete" rule guarantees unbounded growth of stale failure context.

**Failure scenario:** A flaky test causes 12 VERIFICATION_FAIL cycles. `.progress.md` accumulates 12 nearly-identical failure blocks. The next agent reading this file wastes context window on redundant information. On a long-running spec, `.progress.md` becomes so large it degrades agent performance.

**Fix:** Allow deduplication of consecutive identical failures. Or require a structured format where the latest result for a given task *replaces* the previous one, while preserving the history of *different* tasks.

---

## 10. "Cross-CLI Portability" Is Claimed but Not Achieved

The spec says failure reports must be "self-contained enough for another executor to retry without asking what happened." But the required output format is:

```
- **Expected**: [what should happen]
- **Actual**: [what happened]  
- **Fix hint**: [suggested remediation]
```

This is natural language. It's not a command. It's not a patch. A different CLI agent reading "Fix hint: ensure the auth middleware checks token expiry" has to *interpret* that, which is exactly what "without asking what happened" was supposed to avoid.

**Fix:** Require the failure output to include the *exact command to reproduce the failure* and, where possible, the *exact file + line* that needs to change. "Fix hint" should be renamed "Reproduction command" with a runnable shell command.

## Round 1 — DEFENDER

# Defense: `qa-engineer` Agent Specification

## The Concern the Critic Missed

Before addressing the ten points: the critic focused almost entirely on **mechanical edge cases and input validation** while missing the most important design vulnerability in this specification:

**The agent has no mechanism to validate that the Verify commands themselves actually test the implementation.** The entire QA gate depends on the quality of the `[VERIFY]` task block written by the `task-planner` agent. If the planner writes `echo "ok" && exit 0` as the verify command, this QA agent will dutifully run it, get exit code 0, and stamp `VERIFICATION_PASS`. The adversarial posture the spec claims ("you look for what is wrong, not what is right") is structurally undermined because the agent's primary verification mechanism — running the specified commands — is a pure pass-through of someone else's work.

Step 3 (mock anti-pattern scan) partially addresses this for test quality, but there's nothing that asks: "Do these verify commands actually exercise the code that was changed?" A verify command could run a passing test suite that doesn't cover the new feature at all. This is a **false-pass vector at the architectural level**, far more dangerous than any missing-file edge case, because it silently degrades the entire spec-driven pipeline's quality guarantee. The fix: add a Step 1.5 that cross-references the verify commands against the task's "Do" steps to confirm the commands are testing what was just built, not unrelated passing tests.

This matters more than anything the critic raised because every other issue produces a visible failure (crash, hang, contradiction). This one produces **invisible success** — the worst kind of QA failure.

## Point-by-Point Response

### Point 1: No Handling of Missing or Malformed Inputs — PARTIALLY CONCEDE

The critic identifies a real textual gap, but **dramatically overstates the severity to P0** by treating an LLM agent prompt as a deterministic program specification.

This agent runs inside the Ralph Specum orchestration pipeline. It is invoked by the `spec-executor`, which is responsible for providing valid inputs. The `task-planner` creates `.progress.md` as part of its output. The `product-manager` creates `requirements.md`. By the time the QA agent is invoked, these files exist — that's the pipeline's invariant, not this agent's responsibility.

More importantly: **this is an LLM agent, not a bash script.** If it tries to read `.progress.md` and gets a file-not-found error, it can reason about that and handle it. LLMs don't "crash" on missing files — they get an error message from the Read tool and adapt. The critic's crash scenario assumes a rigidity that doesn't exist in this execution model.

**What I concede:** The spec should handle the no-verify-commands case explicitly. A `[VERIFY]` block with no verify commands should be an immediate `VERIFICATION_FAIL`, not a vacuous pass. That's a legitimate gap.

**What I reject:** The P0 severity, the crash scenario, and the need for a formal "Input Validation" step. The orchestrator provides the contract; the agent handles errors through its inherent reasoning capability. Adding a full input validation step adds spec bloat for a scenario that occurs only when the upstream pipeline is already broken — and if the pipeline is broken, a validation step in the QA agent doesn't save you.

**Proposed fix (minimal):** Add one line to Constraints: "If the `[VERIFY]` task block contains no Verify commands, output VERIFICATION_FAIL with a failure noting that no verification was possible."

---

### Point 2: Mock Anti-Pattern Scan Is Unscoped — PARTIALLY CONCEDE

The critic is right that "test files touched by the preceding tasks" is insufficiently specified. However, the proposed failure scenario — the agent "scans nothing" or "scans everything" — ignores that **the agent has full tool access including `git` and can derive the changeset itself.**

The agent can run `git diff --name-only HEAD~1 -- '*.test.*' '*.spec.*'` or examine the task's "Do" steps (which typically reference specific files). An LLM agent reading "Do: Create auth.test.ts with unit tests for the auth module" knows exactly which test files to examine. The critic treats the agent as if it can only follow explicit mechanical instructions, which undersells the entire point of using an AI agent for QA.

**What I concede:** The spec should provide a default mechanism for identifying target files rather than relying on implicit reasoning. The critic's suggested `git diff --name-only` approach is reasonable as a default.

**What I also concede:** The 70% threshold is poorly specified for mechanical verification. An LLM can make a reasonable judgment about mock-heaviness, but framing it as a precise percentage creates false precision.

**What I reject:** The claim that this step is "unexecutable" (the critic's P0 rating). It's under-specified, not impossible. An agent with tool access can figure out which files changed.

**Proposed fix:** Replace "test files touched by the preceding tasks" with "test files identified by running `git diff --name-only HEAD~N` where N is the number of preceding implementation tasks, or files explicitly named in the task's Do steps." Downgrade the 70% rule to: "Flag test files where mock/stub setup substantially outweighs meaningful assertions — use judgment, not a hard threshold."

---

### Point 3: Exit Code Contradiction — CONCEDE

The critic is right. The Constraints section's absolute "Exit code 0 is required" directly contradicts Step 1's "unless the task expects failure output." This is a genuine textual bug that will cause inconsistent behavior on negative tests.

**I accept the critic's proposed fix almost verbatim.** The Constraints line should read:

> "NEVER pass a verification with unexpected exit codes. Exit code 0 is required unless the task explicitly specifies an expected non-zero exit code, in which case the actual exit code must match the expected one exactly."

This is clear, unambiguous, and handles negative test cases correctly.

---

### Point 4: Before/After Comparison Has No Baseline — REJECT

The critic misreads this step. It says:

> "**If** the task specifies a BEFORE/AFTER check (e.g., 'verify file X changed from A to B')"

The task block itself defines what the before and after states are. The example is literal: "verify file X changed from A to B." The agent checks whether file X now contains B. This doesn't require a baseline commit — it requires reading the current file and checking it matches the expected state.

For cases where `git diff` is relevant, the task block would specify the comparison (e.g., "verify the config file differs from main branch"). The agent has git access and can run whatever diff the task requires.

**The critic's failure scenario** — "the executor made the change three commits ago, then made formatting fixes" — is a scenario where the **task block is poorly written**, not where this spec fails. If the task says "verify file X contains Y" and file X contains Y, the check passes regardless of commit history. If the task says "verify git diff shows change Z" without specifying a baseline, that's a task-planner bug.

**No fix needed.** The step is correctly scoped as conditional and task-driven. The baseline comes from the task definition, not from this spec.

---

### Point 5: `.progress.md` Race Condition and Append-vs-Mutate — PARTIALLY CONCEDE

**Race condition: REJECT.** The Ralph Specum pipeline is sequential by design. The spec-executor runs a task, then invokes the QA engineer for verification. If verification fails, the executor retries. These are not concurrent operations — the orchestration layer serializes them. The critic assumes concurrent access that doesn't exist in the actual system architecture.

**Append-vs-mutate contradiction: CONCEDE.** The critic correctly identifies that "Set Current Task to X" is a mutation while the rule says "Append only." These are contradictory instructions, and the agent will handle this inconsistently.

**Proposed fix:** Replace the conflicting instructions with:

> "**Learnings section:** Append only. Never delete existing learnings.
> **Current Task field:** Replace the existing value. This is the one field that is updated in-place, not appended."

This acknowledges the two different operations explicitly rather than trying to force both under "append only."

---

### Point 6: No Timeout or Resource Bounds — REJECT

The critic ignores the execution environment. This agent runs inside Claude Code, where **the Bash tool has a built-in 120-second default timeout (configurable up to 600 seconds).** A hanging `docker-compose up` will be killed by the tool's timeout, not left running forever. The agent doesn't need to implement its own timeout because its execution environment already provides one.

The destructive command concern (`rm -rf /`, `git push --force`) is similarly moot:
- The agent runs in a sandboxed environment with restricted permissions
- The verify commands come from `tasks.md`, written by the task-planner agent within the spec pipeline — not from untrusted external input
- The user's own rules (`.claude/rules/soul.md`) already specify: "Don't run code you don't understand"

**The critic's failure scenario requires:** (a) a task-planner that writes destructive verify commands, (b) an execution environment with no timeouts, and (c) no sandboxing. None of these hold in the actual deployment context.

**No fix needed.** The execution environment provides the guardrails. Duplicating them in the agent spec would be redundant and would fall out of sync with the actual environment's capabilities.

---

### Point 7: AC Check Method Undefined — PARTIALLY CONCEDE

The critic frames this as "redundant with Step 1 or impossible to execute." I'd argue it's **neither** — it's the step that justifies using an AI agent instead of a shell script.

The whole point of Step 2 is to catch the gap between "the commands passed" and "the intent was satisfied." A `npm test` passing tells you tests pass; it doesn't tell you whether AC-1.3 ("Error messages must be user-friendly") is actually met. The LLM agent can **read the implementation code**, check the test coverage against the AC's intent, and flag gaps. That's precisely the kind of judgment call that an AI QA agent is suited for and that a mechanical check cannot provide.

**What I concede:** The step should specify *how* the agent should investigate. Right now it says "check the actual implementation" without saying what tools to use.

**Proposed fix:** Expand Step 2 to:

> "For each AC-X.Y referenced in the task:
> 1. Read the AC definition from requirements.md
> 2. Identify the implementation files from the task's Do steps
> 3. Read the relevant source code and test files
> 4. Verify that the implementation satisfies the AC's intent, not just that tests pass
> 5. If the AC cannot be verified from code inspection alone (e.g., requires manual testing, visual inspection), note it as 'AC-X.Y: Not mechanically verifiable — requires manual check' and do NOT count it as a pass or fail"

This preserves the value of AI-driven AC validation while being explicit about the method and honest about its limits.

---

### Point 8: Signal Parsing Fragility — PARTIALLY CONCEDE

The concern is valid in principle — LLMs can produce the signal words in explanatory text. However, the spec already addresses this more than the critic acknowledges:

- The `<mandatory>` block says "Your response MUST end with EXACTLY one of these two signals"
- The Constraints section says "Output MUST contain EXACTLY one of: VERIFICATION_PASS or VERIFICATION_FAIL"
- The output templates show the signal as a standalone final line

In practice, the consuming agent (spec-executor) parses the **last line** of output, not a regex scan of the full text. The critic's scenario — "Unlike the previous run which ended in VERIFICATION_FAIL..." — would only cause ambiguity with a naive full-text regex, not with last-line parsing.

**What I concede:** Adding an explicit "do not use these words elsewhere in the output" constraint is cheap insurance and worth adding.

**What I reject:** The suggestion to switch to XML comments or JSON blocks. The current format is human-readable, which matters for debugging. Adding structural wrapping adds complexity without proportional benefit given that last-line parsing resolves the ambiguity.

**Proposed fix:** Add to Constraints: "Do NOT use the strings VERIFICATION_PASS or VERIFICATION_FAIL anywhere in your output except as the final signal line."

---

### Point 9: No Idempotency Guarantee — REJECT

The critic's scenario — 12 retry cycles producing 12 duplicate failure blocks — treats a symptom as the disease. If you're hitting 12 retries of the same verification, the problem isn't `.progress.md` bloat; it's that the implementation is fundamentally broken or the task is ill-specified.

The append-only design is **intentional and correct** for this use case:

1. **Failure entries are not always identical.** Each retry may fail differently as the executor attempts fixes. Deduplication would lose this progression.
2. **The history IS the value.** When debugging why a task took 5 retries, the learnings log is the forensic record. Replacing entries destroys diagnostic information.
3. **Context window bloat is a theoretical concern.** In practice, specs have 5-15 tasks. Even with aggressive retries, `.progress.md` stays well under 10KB — negligible against a 200K+ context window.

The critic provides no evidence that this is a **likely** failure in practice, only that it's **possible**. By their own standard (scenario + affected party + likelihood), this doesn't meet the bar for concession.

**No fix needed.** The append-only design serves its purpose. If a future deployment shows actual context window pressure from `.progress.md`, it can be addressed then — "medir antes de optimizar" (Principle 7).

---

### Point 10: Cross-CLI Portability Is Aspirational — REJECT (HARD)

The critic fundamentally misunderstands the consumer of this output. The failure report is consumed by **another LLM agent** (the spec-executor), not by a shell script or CI system. The critic says:

> "A different CLI agent reading 'Fix hint: ensure the auth middleware checks token expiry' has to *interpret* that"

Yes. That's the point. The consumer is an AI agent that excels at interpretation. The spec already requires:

- **Exact failing command** — the executor can rerun it
- **Relevant file path(s)** — the executor knows where to look
- **Concrete mismatch between expected and actual** — the executor knows what's wrong

This is more than sufficient for an LLM executor to fix the issue in one pass. Requiring "exact file + line that needs to change" would mean the QA agent is doing the executor's job. And requiring a "reproduction command" is redundant — the failing verify command IS the reproduction command.

The critic wants machine-parseable output for a machine-interpretable consumer. Natural language IS the machine-parseable format when the machine is an LLM.

**No fix needed.** The output format is well-matched to its actual consumer.
