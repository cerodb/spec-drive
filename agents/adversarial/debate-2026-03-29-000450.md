## Round 1 — CRITIC

# Critique: Product Manager Agent Specification

**Content type:** Agent prompt / process specification for an AI sub-agent in a spec-driven development pipeline.
**"Improved" means:** Unambiguous instructions that produce consistent, complete, portable `requirements.md` files across any spec — with no silent failures, no dropped context, and no room for the agent to hallucinate or drift.

---

## Critical Issues (ordered by damage potential)

### 1. No Validation of Input Quality — Garbage In, Garbage Out

The spec says both `idea.md` and `research.md` are mandatory and checks only for *existence*. It never instructs the agent to validate that these files contain the expected sections (`Vision`, `Constraints`, `Executive Summary`, `Feasibility Assessment`, `Open Questions`).

**Failure scenario:** `idea.md` exists but is a stub with just a title and no `Constraints` section. The agent proceeds, produces an NFR table with zero entries (no constraints to pull from), and a hollow Out of Scope section. The architect downstream receives a `requirements.md` that *looks* complete but is structurally empty. Nobody catches it until implementation.

**Fix:** Add explicit structural validation. List the required sections/headings in each input file. If a required section is missing or empty, stop and report *which* section is missing, not just which file.

---

### 2. Feature Area ↔ User Story Is a Forced 1:1 Mapping That Will Break

> "Each feature area becomes a user story."

This is architecturally wrong. A feature area (e.g., "Authentication") routinely decomposes into multiple user stories (login, logout, password reset, session management). Forcing 1:1 means the agent must either:
- Cram multiple capabilities into one bloated story (violating INVEST principles), or
- Artificially split feature areas to match story count (producing incoherent groupings).

**Failure scenario:** A feature area "Data Export" maps to one user story covering CSV export, PDF export, scheduled exports, and export history. The acceptance criteria balloon to 15+ items on a single story. The task planner downstream can't decompose it into meaningful work units.

**Fix:** Allow N user stories per feature area. Change the instruction to: "Each feature area may contain one or more user stories. Split when a single story would require more than 5-7 acceptance criteria."

---

### 3. Priority System Is Undefined and Subjective

The FR table uses `High/Medium/Low` priority but the only guidance is:

> "Priority reflects impact on the core vision from idea.md"

No rubric. No definition of what separates High from Medium. No instruction on distribution (can everything be High?).

**Failure scenario:** The agent marks 90% of FRs as "High" because everything in `idea.md` sounds important to the vision. The downstream task planner has no meaningful way to sequence work. Priority becomes noise.

**Fix:** Define the levels concretely. Example: High = project fails without it (MVP-blocking). Medium = significantly degrades value if missing. Low = nice-to-have, deferrable. Add a constraint: "No more than 40% of FRs should be High priority. If you exceed this, re-evaluate against the MVP threshold."

---

### 4. NFR Targets Are Demanded but Source Material Likely Doesn't Contain Them

> "Every NFR must be measurable with a specific target (e.g., 'response time < 200ms')"

The inputs are `idea.md` (vision/constraints) and `research.md` (feasibility). Neither of these typically contains quantified performance targets. The agent is simultaneously told "do NOT invent requirements that aren't grounded in idea.md or research.md" AND "every NFR must have a specific measurable target."

**Failure scenario:** `idea.md` says "should work on low-end hardware." The agent must either invent a specific target (violating the grounding constraint) or omit the NFR (violating the measurability mandate). It's a double-bind. In practice, the agent will hallucinate plausible-sounding numbers like "< 500ms on 2-core CPU" with no basis.

**Fix:** Add an escape hatch: "If the source material doesn't specify a concrete threshold, note the NFR with `[TARGET TBD — requires stakeholder input]` and flag it in the Dependencies section as a blocking decision."

---

### 5. Success Criteria Must "Reference Specific FR/NFR IDs" — But IDs Don't Exist Yet When Thinking About Success

The Success Criteria step says:

> "Reference specific FR/NFR IDs."

But the agent is writing all of this in a single pass. The cognitive load instruction is: define stories → define FRs → define NFRs → then write success criteria referencing them. This works *if* the agent doesn't need to revise anything. But success criteria often reveal that an FR is missing or an NFR target is wrong.

**Failure scenario:** The agent writes 12 FRs, then writes a success criterion "All data exports complete within SLA" and realizes there's no FR for export SLA. It either backtracks and renumbers (potentially breaking AC references) or writes a success criterion that references a non-existent FR. The downstream consumer sees a dangling reference.

**Fix:** Either (a) instruct the agent to do a review pass after drafting all sections, verifying all cross-references resolve, or (b) explicitly permit adding FRs during the Success Criteria step with a note that they were derived from goal validation.

---

### 6. The 3000-Word Limit Is Arbitrary and Creates a Silent Quality Cliff

> "Keep the document under 3000 words — concise requirements are usable requirements"

For a 2-story project, 3000 words is generous. For a 15-story project with complex domain logic, 3000 words forces the agent to silently cut content. There's no instruction on *what* to cut or how to signal that the limit was hit.

**Failure scenario:** A complex spec generates 12 user stories. At story 9, the agent hits the word limit. It truncates the last 3 stories' acceptance criteria to single-line summaries. The architect receives a document where some stories have rigorous ACs and others have vague hand-waves. Nobody realizes stories 10-12 are underspecified until implementation fails verification.

**Fix:** Either (a) make the limit proportional ("~250 words per user story"), or (b) mandate that if the limit forces truncation, the agent must add a `## Truncation Notice` section listing what was cut and why, or (c) allow the agent to split into `requirements.md` and `requirements-extended.md`.

---

### 7. AC-X.Y Numbering Is Brittle Across Revisions

AC IDs are coupled to story numbers (`AC-1.1`, `AC-2.3`). If stories are reordered, merged, or deleted during the `refactor` phase, every downstream reference to AC IDs breaks — in the design doc, in the task list, in test plans.

**Failure scenario:** During refactoring, US-3 is merged into US-2. AC-3.1 through AC-3.4 either get renumbered (breaking references in `tasks.md` and `design.md`) or left orphaned. The QA engineer verifying against AC-3.2 finds nothing matching in the revised requirements.

**Fix:** Either (a) use stable IDs decoupled from story order (e.g., `AC-AUTH-01`), or (b) mandate that the refactor-specialist must update all downstream references when AC IDs change, with an explicit checklist.

---

### 8. No Conflict Resolution Between idea.md and research.md

Both are "source of truth," but they can contradict each other. `idea.md` might say "must support offline mode." `research.md` might conclude "offline mode is infeasible with current architecture."

**Failure scenario:** The agent encounters a contradiction, has no instruction on which source takes precedence, and silently picks one — likely `idea.md` because it's listed first. It writes a requirement for offline mode. The architect designs for it. Research said it was infeasible. Wasted sprint.

**Fix:** Add a conflict resolution protocol: "If research.md contradicts idea.md on feasibility, flag it as a `## Unresolved Conflicts` section. Do NOT resolve the conflict by choosing a side. Surface it for human decision."

---

### 9. Glossary Placement Guarantees It Will Be Ignored

The Glossary is the last section of the output and the last step in the process. By the time the agent gets there, it has already used domain terms throughout the document without definition.

**Failure scenario:** The requirements use "hydration" to mean server-side data pre-loading. The architect reads it as React hydration. The glossary exists at the bottom but the architect never scrolls that far. Misinterpretation propagates through design and implementation.

**Fix:** Either (a) move glossary to immediately after User Stories (before the technical tables), or (b) instruct the agent to use inline definitions on first use *in addition to* the glossary, e.g., "hydration (server-side data pre-loading into cache)".

---

### 10. `.progress.md` Update Is Append-Only With No Structure

> "Append only. Never delete existing learnings."

There's no format specified for what gets appended. No timestamp. No attribution. After 5 phases of appending, `.progress.md` becomes an unstructured wall of text with no way to distinguish which phase produced which learning.

**Failure scenario:** The task planner reads `.progress.md` and finds 30 bullet points of "learnings" with no dates or phase markers. It can't tell which are from research (still relevant) vs. which are from a requirements draft that was later revised (potentially stale).

**Fix:** Specify the append format: `### [Phase] — [ISO timestamp]\n- Learning 1\n- Learning 2`. This makes the file scannable and allows downstream agents to filter by phase.

---

### 11. "Verification" Column in FR Table Is Underspecified

The FR table requires a "Verification" column with values like "unit test, integration test, manual check, CLI command." But there's no guidance on granularity or who performs verification.

**Failure scenario:** Every FR gets "manual check" as verification because it's the easiest to write. The QA engineer receives a requirements doc where nothing is automated-testable. The entire verification plan becomes "someone clicks through it," which is exactly what spec-driven development is supposed to prevent.

**Fix:** Add a constraint: "At least 60% of FRs must have automated verification methods (unit test, integration test, CLI command). If a requirement can only be verified manually, justify why in the Verification column."

---

### 12. No Handling of Partial Failures or Interruption

The agent reads two files, performs 7 steps, writes one output file, then updates `.progress.md`. If it fails at step 5, nothing is written. There's no checkpointing, no partial output, no error reporting to `.progress.md`.

**Failure scenario:** The agent hits a context limit or crashes mid-generation. No output is written. No error is logged. The orchestrator sees no `requirements.md` and no error in `.progress.md`. It either retries blindly or stalls.

**Fix:** Instruct the agent to write `.progress.md` with status `"In Progress"` *before* starting generation, then update to `"Complete"` after. If the agent dies mid-execution, the stale "In Progress" status signals the failure to the orchestrator.

## Round 1 — DEFENDER

# Defense: Product Manager Agent Specification — Round 1

## The Concern the Critic Missed

Before addressing the twelve points: **the critic never raised the absence of source traceability in the output.**

The spec mandates "Every FR must trace back to at least one user story" (FR → US), but there is **zero mechanism for tracing user stories or requirements back to specific passages in `idea.md` or `research.md`**. The document simultaneously demands "Do NOT invent requirements that aren't grounded in idea.md or research.md" — but provides no audit trail to verify compliance.

This matters more than input validation (point #1) or conflict resolution (point #8) because it affects **every single requirement produced**. When the architect questions "where did FR-7 come from?", nobody can verify it without re-reading both source documents end to end. When the refactor-specialist updates requirements, they can't tell which source material anchors which requirement. When a stakeholder challenges a priority, there's no citation to point to.

A simple fix: require each user story to include a `**Source:**` line referencing the section of `idea.md` or `research.md` it derives from. Lightweight, auditable, and it forces the agent to ground every story explicitly.

This is more important than anything the critic raised because it's a **systemic integrity gap** — it undermines the core promise of the spec-driven workflow (traceability from goal to implementation).

## Responses to the Critic's Points

### Point 1: Input Validation — PARTIAL CONCEDE

The critic demonstrates a real scenario: a stub `idea.md` produces structurally hollow output. The architect downstream is affected. I concede the problem exists.

However, the proposed fix — listing every required section and halting on any missing one — is **too rigid**. This agent operates within the ralph-specum pipeline, where `idea.md` is produced by the research-analyst through a structured interview framework. A stub reaching this stage is an orchestration failure, not a product-manager-agent failure. Making this agent halt-and-report for upstream problems is appropriate; making it enforce a specific heading schema makes it brittle to any formatting variation in the upstream agents.

**Better fix:** Add a validation step that checks for *minimum viable content*, not specific headings:

> "After reading both files, verify each contains at least 200 words of substantive content. If either file appears to be a stub or placeholder (fewer than 200 words, no discernible structure), stop and report: '[filename] appears incomplete — cannot produce grounded requirements.' If a file exists but lacks an expected section (Vision, Constraints, Feasibility), note the gap in your output's Dependencies section as a blocking input deficiency rather than halting entirely."

This catches the stub problem without coupling the agent to a rigid heading schema that may evolve.

### Point 2: 1:1 Feature-to-Story Mapping — CONCEDE

The critic is right. "Each feature area becomes a user story" is unnecessarily rigid, and the failure scenario (bloated stories with 15+ ACs) is realistic for any non-trivial spec.

I accept the critic's fix with a minor refinement. Change the instruction to:

> "Each feature area produces one or more user stories. If a single story would require more than 7 acceptance criteria, decompose it into smaller stories within the same feature area."

The 5-7 range the critic suggested is reasonable; I'd pin it at 7 to avoid premature splitting on stories that are simply thorough.

### Point 3: Priority Rubric — PARTIAL CONCEDE

The problem is real: undefined priority levels reduce the signal value of the column. I concede that a rubric would help.

However, the critic's proposed distribution cap ("no more than 40% High") is **the wrong mechanism**. A 3-FR project where all 3 are genuinely MVP-critical would force the agent to artificially downgrade one. Distribution caps create perverse incentives — the agent game-theorizes the cap rather than assessing impact honestly.

**Better fix:** Define the levels without a cap:

> "**High** = The project cannot deliver its core value without this. Removal would invalidate the vision in idea.md. **Medium** = Significantly enhances value but the core proposition survives without it. **Low** = Additive improvement, deferrable to a future iteration. If all FRs are High, re-examine whether the project scope in idea.md is too broad and note this in Dependencies as a scope concern."

The last sentence catches the "everything is High" failure mode without an arbitrary percentage.

### Point 4: NFR Target Double-Bind — CONCEDE

This is the critic's strongest point. The contradiction between "don't invent" and "every NFR must have a specific target" is a genuine double-bind that will reliably produce hallucinated numbers. The likelihood is high — most `idea.md` files contain qualitative constraints ("should be fast", "must work on modest hardware"), not quantified thresholds.

I accept the proposed fix. Add:

> "If the source material specifies a qualitative constraint without a numeric threshold, write the NFR with `[TARGET TBD — requires stakeholder input]` as the target value and add a corresponding entry to the Dependencies section noting the decision needed."

This preserves the requirement's existence (it IS a real requirement) without forcing fabricated numbers.

### Point 5: Success Criteria Cross-References — REJECT

The critic's concern assumes the agent writes linearly and can't backtrack. But this is an LLM generating a single document — it plans the full structure before writing, and routinely produces documents with internally consistent cross-references. The scenario of "realizes there's no FR for export SLA" at step 6 is not how generation works in practice: the agent has already conceptualized all requirements before committing any section to text.

Moreover, the spec already handles this implicitly. If the agent identifies a gap during success criteria formulation, it adds the FR during step 4 (it's generating the full document, not committing sections sequentially to disk). The "single pass" here is a single output file, not a sequential commit of immutable sections.

Adding a "review pass" instruction adds cognitive overhead to the prompt for a problem that doesn't materially occur in practice. **No change needed.**

### Point 6: 3000-Word Limit — REJECT

The critic frames the limit as arbitrary. It isn't — it's a **design constraint** serving the spec's core value: portability and usability. A 5,000-word requirements document is one that nobody reads completely. The limit forces discipline.

The critic's failure scenario (12 stories, truncation at story 9) reveals a **scope problem**, not a word-limit problem. A spec with 12 user stories is too large for a single spec in this workflow. The ralph-specum pipeline is designed for focused, bounded projects. If the agent is producing 12 stories, the correct response is to note in Dependencies: "Scope exceeds single-spec capacity — recommend splitting into sub-specs."

The truncation-notice suggestion is a reasonable safety valve, though. I'll accept that narrow addition:

> "If the document approaches the word limit before all stories are fully specified, add a `## Scope Advisory` note recommending the project be split into sub-specs, and prioritize complete coverage of High-priority stories over partial coverage of all stories."

This is better than the critic's suggestion of a second file (which breaks the single-artifact contract downstream agents expect) or proportional limits (which are harder to enforce and still arbitrary).

### Point 7: AC-X.Y Numbering Brittleness — REJECT

The critic describes a real problem — but attributes it to the **wrong agent**. This spec defines the *product manager* agent, which produces the initial `requirements.md`. ID stability during the refactor phase is the responsibility of the `refactor-specialist` agent (which the skill list explicitly includes: "update spec files methodically after execution").

The AC-X.Y format is **correct for initial generation**: it's simple, unambiguous, and creates a clear hierarchy (story → criteria). Switching to semantic IDs like `AC-AUTH-01` requires the agent to invent stable semantic prefixes, which is a naming problem that adds complexity without clear benefit at generation time.

If downstream ID stability is a concern, the fix belongs in the refactor-specialist's spec: "When renumbering or merging user stories, produce an ID mapping table and update all references in design.md and tasks.md." That's where the problem lives; that's where the fix should go.

### Point 8: Conflict Resolution — CONCEDE

The critic correctly identifies that two documents declared as "source of truth" can contradict each other, and the agent has no protocol for this. The scenario (idea.md demands offline mode, research.md says it's infeasible) is realistic — the research phase exists precisely to test idea.md's assumptions, so contradictions are *expected*, not edge cases.

I accept the proposed fix. Add an `## Unresolved Conflicts` section to the output format:

> "If research.md contradicts or qualifies a goal in idea.md (e.g., a desired capability is assessed as infeasible), do NOT silently choose one source. Add an `## Unresolved Conflicts` section listing each contradiction with references to both files. These require human decision before the architect proceeds."

This is clean, non-opinionated, and surfaces conflicts without blocking the entire document.

### Point 9: Glossary Placement — REJECT

The critic claims the glossary "guarantees it will be ignored" because it's last. This misrepresents how technical documents work. Glossaries appear at the end of IEEE standards, RFCs, ISO specs, and virtually every requirements document in professional practice. Readers of technical specifications know to check the glossary — it's a reference section, not a narrative section.

The failure scenario — "the architect never scrolls that far" — is an indictment of the architect, not the document structure. If an architect reads half a requirements document and starts designing, no amount of glossary repositioning fixes that problem.

The inline-definition suggestion has superficial appeal but would **bloat the document** (worsening the critic's own concern about the 3000-word limit) and create maintenance headaches: every inline definition must stay synchronized with the glossary entry across refactoring cycles.

**The glossary stays where it is.** If specific projects have high domain-ambiguity risk, the glossary will naturally be more detailed — but its placement is correct.

### Point 10: .progress.md Append Format — CONCEDE

Fair point. The append-only instruction without a format produces an unstructured log that degrades with each phase. The critic's proposed format is sensible and minimal.

**Fix:** Add to the Progress Update section:

> "Append in this format:\n```\n### Requirements — [ISO timestamp]\n- [Learning 1]\n- [Learning 2]\n```"

Simple, scannable, filterable by phase. Accepted.

### Point 11: Verification Column — REJECT

The critic proposes "at least 60% of FRs must have automated verification methods." This is an **arbitrary constraint that doesn't account for project type**.

A CLI tool naturally has 90%+ automated verification. A design system with visual components might legitimately have 70% manual verification (visual regression, layout inspection). An API integration project falls somewhere between. Imposing a fixed percentage creates a perverse incentive: the agent labels verifications as "unit test" to meet the quota, even when the honest answer is "manual check."

The existing spec already constrains verification to specific categories ("unit test, integration test, manual check, CLI command"). The agent must pick one and describe it concretely. That's sufficient — the task planner and QA engineer downstream can escalate if the verification plan is inadequate for their context.

The "justify why" suggestion for manual checks has merit in isolation but adds word count to every manually-verified FR, compounding the word-limit pressure the critic already flagged. **No change.**

### Point 12: Partial Failure Handling — REJECT

The critic wants the agent prompt to include checkpointing logic (write "In Progress" before starting, update to "Complete" after). This conflates **orchestration concerns with agent concerns**.

The ralph-specum pipeline has an orchestrator (the spec-executor) that manages agent lifecycle, detects failures, and handles retries. Adding failure-detection instrumentation to individual agent prompts:

1. **Bloats every agent spec** with identical boilerplate
2. **Duplicates responsibility** that belongs in the orchestration layer
3. **Doesn't actually help** — if the agent crashes mid-write, it may also crash before updating `.progress.md`, making the "In Progress" signal unreliable anyway

The correct fix is in the orchestrator: "If the product-manager agent returns without producing `requirements.md`, log the failure and retry or escalate." That's where timeout detection, crash recovery, and status tracking belong.

# Round 2 — CRITIC

## Attacking the Defender's Fixes and Rebuttals

---

### A. The "200-Word Minimum" Validation Is Trivially Gameable and Misses the Point (Point #1 Fix)

The defender replaced structural validation with:

> "verify each contains at least 200 words of substantive content"

This is worse than the original problem. 200 words of *wrong* content passes the check. An `idea.md` that contains 300 words about the company's founding history and zero words about vision or constraints sails through. The word-count proxy measures *volume*, not *structure*. The defender's own argument — "a stub reaching this stage is an orchestration failure" — contradicts adding this check at all. If it's not this agent's job to validate upstream, why add a validation step? And if you *are* adding one, make it meaningful.

The defender also proposes noting missing sections in Dependencies rather than halting:

> "note the gap in your output's Dependencies section as a blocking input deficiency rather than halting entirely"

This creates a **zombie document**: `requirements.md` exists, looks complete, but contains a buried note in section 6 saying "by the way, the input was deficient." The downstream architect sees `requirements.md` exists → proceeds. The deficiency note is a passive landmine.

**The fix that actually works:** Check for the presence of *any* content under headings that contain keywords like "vision," "goal," "constraint," or "scope" in `idea.md`, and "feasibility," "finding," or "assessment" in `research.md`. Not rigid heading names — keyword scanning. If nothing matches, halt. If partial matches, proceed but add a **top-of-document warning banner**, not a buried dependency note:

```markdown
> ⚠️ INPUT GAPS: idea.md lacks identifiable constraints section. 
> NFRs and Out of Scope may be incomplete. Review before proceeding.
```

Visible. Impossible to miss. Doesn't halt on formatting variation.

---

### B. The 7-AC Cap Creates a New Failure Mode (Point #2 Fix)

The defender accepted multi-story feature areas but pinned decomposition at 7 ACs:

> "If a single story would require more than 7 acceptance criteria, decompose it into smaller stories"

This creates a **perverse decomposition incentive**. The agent doesn't split stories based on *cohesion* (which stories make sense as independent units of value); it splits based on *AC count* (an arbitrary arithmetic trigger). Result: stories that are semantically incoherent but happen to have ≤7 ACs each.

**Scenario:** Feature area "User Authentication" naturally has 9 ACs across login, session management, and password reset. The agent hits the 7-AC trigger and splits it. But where? It might split "login + session" (6 ACs) and "password reset" (3 ACs) — reasonable. Or it might split "login + password reset" (5 ACs) and "session management" (4 ACs) — incoherent grouping that confuses the task planner. The spec gives zero guidance on *how* to split, only *when*.

**Better fix:** Replace the mechanical threshold with a semantic heuristic:

> "Each feature area produces one or more user stories. Split when a single story covers multiple distinct user workflows or when acceptance criteria address unrelated capabilities. Aim for stories that can be implemented and verified independently."

This produces stories that make sense as work units, regardless of AC count.

---

### C. The Priority Rubric's Escape Hatch Undermines Itself (Point #3 Fix)

The defender defined:

> "**High** = The project cannot deliver its core value without this. Removal would invalidate the vision in idea.md."

Then added:

> "If all FRs are High, re-examine whether the project scope in idea.md is too broad and note this in Dependencies as a scope concern."

This is contradictory. If the agent honestly applies the rubric and every FR genuinely meets the "removal would invalidate the vision" bar, the instruction says to *question its own correct assessment* and flag a scope problem. But maybe the project just has 4 FRs and they're all critical. The agent can't distinguish "scope is too broad" from "this is a tightly-scoped MVP where everything is essential."

More importantly: "note this in Dependencies as a scope concern" is another **buried passive note**. The task planner downstream doesn't read Dependencies looking for meta-commentary about priority distribution. It reads the FR table and sees all-High. The note is decorative.

**Better fix:** Keep the rubric definitions (they're good). Drop the all-High escape hatch. Instead, add a **MoSCoW mapping instruction**:

> "If prioritization produces more than 5 High-priority FRs, re-express using MoSCoW (Must/Should/Could/Won't) to provide finer granularity. Must = cannot ship without. Should = expected but survivable if cut. Could = valuable if time permits. Won't = explicitly deferred."

This gives the downstream consumer *more information*, not a vague scope warning.

---

### D. The Defender's Rejection of Point #5 (Cross-References) Relies on an Untested Claim

> "This is an LLM generating a single document — it plans the full structure before writing, and routinely produces documents with internally consistent cross-references."

This is empirically false for complex documents. LLMs generate tokens sequentially. They do not "plan the full structure before writing" — they predict the next token given preceding context. For a 2500-word document with 10+ FRs and cross-referenced success criteria, dangling references are not theoretical; they are **a known failure mode of long-form LLM generation**.

The defender offers no evidence that this doesn't occur in practice. "It routinely produces consistent cross-references" is an assertion, not a measurement. Has anyone audited 20 outputs from this agent for reference consistency? The spec doesn't include any self-verification step, so errors propagate silently.

**The cost of the fix is near-zero.** Add one sentence:

> "After drafting all sections, verify that every FR/NFR ID referenced in Success Criteria exists in the corresponding table. If a gap is found, add the missing requirement."

One sentence. No cognitive overhead. Catches a real class of errors. The defender's rejection optimizes for prompt brevity over output correctness — the wrong tradeoff for a spec that will be consumed by automated agents downstream.

---

### E. The Defender's Rejection of Point #7 (AC Numbering) Creates a Cross-Agent Coordination Gap

The defender says:

> "ID stability during the refactor phase is the responsibility of the refactor-specialist agent"

This is technically correct but **practically dangerous**. The product-manager agent *chooses* the ID scheme. That choice has downstream consequences. Choosing a scheme that is *inherently* brittle to reordering, then declaring the brittleness someone else's problem, is architecture by abdication.

The defender argues semantic IDs like `AC-AUTH-01` "require the agent to invent stable semantic prefixes." But the agent is *already inventing* feature area names (Step 1). If it identifies a feature area called "Authentication," `AC-AUTH-` is trivially derived. The cognitive cost is near-zero; the downstream stability benefit is significant.

Moreover, the defender's proposed fix — "the refactor-specialist must produce an ID mapping table" — doesn't exist yet. It's a fix for a spec that hasn't been written, for an agent whose behavior isn't defined here. The product-manager agent is shipping a known fragility and hoping a future agent will compensate. That's a bet, not a design.

**Concrete improvement:** Use feature-area-derived prefixes:

> "AC IDs use the format `AC-[AREA]-[N]` where AREA is a short abbreviation of the feature area name (2-5 chars) and N is sequential within that area. Example: `AC-AUTH-1`, `AC-EXPORT-3`."

Stable across reordering. Trivial to generate. Meaningful to readers.

### F. The Defender's Rejection of Point #9 (Glossary) Ignores the Actual Consumer

> "Readers of technical specifications know to check the glossary"

The consumers of this document are **not human engineers reading an IEEE standard**. They are **other LLM agents** — the architect-reviewer and task-planner. LLM agents process documents top-to-bottom with attention that degrades over context length. A glossary at position 2800 of a 3000-word document is in the lowest-attention zone of the context window.

The defender's rebuttal assumes human reading patterns for a document consumed by machines. This is the wrong threat model.

The inline-definition concern ("would bloat the document") is valid, so I'm not re-proposing that. But moving the glossary to position 2 in the document (after the frontmatter, before User Stories) costs zero words and ensures every downstream agent encounters term definitions *before* encountering term usage.

**Fix:** Move Glossary to immediately after the frontmatter header:

```markdown
# Requirements: [spec name]

## Glossary
[Domain terms]

## User Stories
...
```

Zero word-count impact. Maximum attention-window benefit for LLM consumers.

### G. The Defender's Rejection of Point #12 (Partial Failure) Mischaracterizes the Proposal

The defender argues checkpointing "belongs in the orchestration layer" and that an agent writing "In Progress" might crash before even that write completes.

The first argument proves too much: by this logic, `.progress.md` updates don't belong in any agent spec, since progress tracking is an "orchestration concern." Yet the spec already mandates updating `.progress.md`. The defender is selectively applying the separation-of-concerns argument.

The second argument — "the agent might crash before writing In Progress" — is the nirvana fallacy. "This safety measure doesn't catch 100% of failures, therefore don't add it." A write at the *beginning* of execution that succeeds 90% of the time before a mid-generation crash is strictly better than a write only at the *end* that succeeds 0% of the time when the agent crashes.

**The fix is two lines added to the existing `.progress.md` instruction:**

> "Before starting generation, update `.progress.md` with `Current Task: Generating requirements`. After writing `requirements.md`, update to `Current Task: Awaiting next task`."

This is not "checkpointing logic." It's changing the *order* of a write the spec already requires. Minimal change, meaningful signal.

### H. NEW VULNERABILITY: The Source Traceability Fix Is Insufficient

The defender raised a valid gap (source traceability) and proposed:

> "require each user story to include a `**Source:**` line referencing the section of `idea.md` or `research.md` it derives from"

This traces User Stories but leaves **FRs and NFRs untraced**. The FR table has no source column. An FR can be technically derived from a user story (the spec mandates FR → US traceability) but the *user story itself* might drift from its source. The chain is: source document → user story → FR. If the US → source link is the only one verified, an FR that was added "because it seemed related" to a user story has indirect cover but no direct audit trail.

**Fix:** Add a `Source` column to both the FR and NFR tables:

```markdown
| ID | Requirement | Priority | Source | Verification |
|----|-------------|----------|--------|--------------|
| FR-1 | [Description] | High | idea.md §Vision | [How to verify] |
```

Direct traceability at every level. The architect can verify any requirement against its origin without chasing through user story indirection.

### I. NEW VULNERABILITY: The Unresolved Conflicts Section Has No Downstream Protocol

The defender accepted adding `## Unresolved Conflicts`. But what happens to it? The spec says the architect-reviewer reads `requirements.md` as input. There's no instruction in *this* spec about what the architect should do when encountering unresolved conflicts. And more critically, there's no instruction in *this* spec about what happens if the architect proceeds anyway.

**Scenario:** Product manager writes 3 unresolved conflicts. The pipeline continues. The architect reads requirements.md, ignores the conflicts section (it's not referenced in the architect-reviewer's mandatory input sections), and designs around the `idea.md` version. The conflicts were surfaced — and then walked right past.

**Fix:** Add to the output instructions:

> "If the document contains an `## Unresolved Conflicts` section with any entries, add a `BLOCKED` status to the frontmatter: `status: blocked-on-conflicts`. The downstream architect agent must not proceed until conflicts are resolved and this status is removed."

This makes conflicts *mechanically blocking*, not just informational.
