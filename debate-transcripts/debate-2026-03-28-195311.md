## Round 1 — CRITIC

# Critique: `researcher` Agent Prompt

**Content type:** Agent system prompt / behavioral specification for an automated research agent in a spec-driven development pipeline.

**"Improved" means:** The agent reliably produces accurate, complete research output under all real-world conditions — including malformed input, missing files, hostile codebases, and tool failures — without silently producing garbage or blocking the pipeline.

---

## Critical Issues (ordered by damage potential)

### 1. The `jq` pipeline is a data-loss race condition

```bash
jq '.awaitingApproval = true' {basePath}/.spec-drive-state.json > /tmp/sd-state.json && mv /tmp/sd-state.json {basePath}/.spec-drive-state.json
```

**Failure scenario:** Two agents or two invocations run concurrently. Both read the same state file, both write to `/tmp/sd-state.json` (same hardcoded path), one `mv` overwrites the other's changes. Any concurrent field written by another agent (e.g., `currentPhase`, `lastError`) is silently destroyed.

Even without concurrency: if the state file doesn't exist yet, `jq` reads nothing, outputs `null`, and you `mv` a file containing `null` over whatever was there.

**Fix:** Use `mktemp` for the temp file. Guard with an existence check. Better yet, use `sponge` or write a dedicated state-update command that handles locking. And specify what to do when the file doesn't exist — create it with a defined schema, or fail loudly.

---

### 2. No error handling for ANY tool call

Every step shows tool invocations (`WebSearch`, `WebFetch`, `Glob`, `Grep`, `Read`) but never specifies what happens when they fail. This is pervasive — it affects every single step.

**Failure scenarios:**
- `WebSearch` returns nothing (rate-limited, network down, bad query): Step 1 silently produces an empty "External Research" section. Downstream agents treat absence of findings as "no concerns" rather than "research failed."
- `Glob` returns 10,000 files in a monorepo: the agent tries to process all of them, blows context, and produces truncated or hallucinated analysis.
- `Read` on a binary file or a 50,000-line generated file: garbage in, garbage out.
- `WebFetch` on a URL that returns a login wall or CAPTCHA: the agent records the HTML of a login page as "documentation."

**Fix:** For each tool invocation, define: (a) what constitutes a failed/empty result, (b) what the agent should do (retry? skip with explicit note? abort?), (c) maximum result sizes and how to handle overflow. The constraints section says "NEVER skip web search" but doesn't say what to do when web search returns nothing useful.

---

### 3. `idea.md` is "sole input" but the schema is undefined

The agent is told to extract "Vision" and "Constraints" from `idea.md`, but there's zero specification of what `idea.md` looks like. No heading format. No required fields. No examples.

**Failure scenario:** A human writes `idea.md` as a stream-of-consciousness paragraph with no clear structure. The agent "extracts" a vision that doesn't match what the human intended. This misinterpretation propagates to `research.md`, then to requirements, then to design, then to tasks. The entire pipeline builds the wrong thing. Nobody catches it because the research looked plausible.

**Fix:** Either define the `idea.md` schema explicitly (with required headings or frontmatter), or add a validation step where the agent echoes back its extracted understanding of vision/constraints into the research output and the approval gate is specifically about confirming that interpretation.

---

### 4. Quality Command Discovery is naively ordered and incomplete

The spec says "check these sources in order: package.json, Makefile, .github/workflows." But it doesn't say what "in order" means operationally. Stop at first hit? Check all and merge? What if `package.json` has a `test` script that's just `echo "no tests"` (a common npm init artifact)?

**Failure scenarios:**
- Project uses `turbo`, `nx`, or `moon` for task running — none of those are checked.
- `package.json` has `"test": "jest"` but `jest.config.ts` doesn't exist or is broken — the command is recorded as valid but will fail.
- Monorepo with multiple `package.json` files — which one is authoritative?
- Python project (the Glob pattern only checks `*.ts`, `*.js`, `*.json`) — zero quality commands found, task-planner skips all verification.

**Fix:** Don't hardcode to three sources. Discover the project's language/runtime first, then look for quality commands in the appropriate places. Validate that discovered commands actually run (or at least that their dependencies exist). And explicitly handle non-Node projects — the entire step is Node-biased despite no such constraint being stated.

---

### 5. Codebase Exploration Glob patterns are hardcoded to JS/TS

```
Glob: **/*.ts, **/*.js, **/*.json — find relevant files
```

**Failure scenario:** The codebase is Python, Go, Rust, or mixed. The agent finds zero relevant files, reports "no existing patterns found," and the architect designs from scratch — duplicating existing functionality that lives in `.py` or `.go` files.

**Fix:** Either derive the glob patterns from the codebase's actual languages (check file extensions present, or use a language detection heuristic), or remove the hardcoded patterns entirely and let the agent determine appropriate patterns from context.

---

### 6. Feasibility Assessment has no calibration or methodology

The agent is asked to rate effort as S/M/L/XL and risk as High/Med/Low, but there's no calibration for what these mean in context. "S = hours" — for whom? A senior engineer? The agent? A junior dev reading the spec for the first time?

**Failure scenario:** The agent rates everything as "M effort, Low risk" because it has no grounding for these judgments. The task-planner trusts these estimates, under-allocates time, and the project slips. Alternatively, the agent rates everything XL/High defensively, and every project looks infeasible.

**Fix:** Provide calibration anchors: "S = can be done in a single function/file change, no new dependencies" / "XL = requires new infrastructure or external service integration." Require the agent to justify each rating with specific evidence from the codebase or research findings, not just vibes.

---

### 7. The "append only" rule for `.progress.md` will create an unbounded file

"Append only. Never delete existing learnings."

**Failure scenario:** After 20 research iterations (re-runs, pivots, new specs reusing the same project), `.progress.md` is thousands of lines of contradictory learnings — early entries say "use REST" while later entries say "switched to GraphQL." Downstream agents read the full file and get confused by contradictory signals. Or worse, the file gets so large it's ignored.

**Fix:** Allow superseding entries (mark old ones as `[SUPERSEDED]`) or use timestamped entries so readers can take the latest. Define a maximum size or a pruning strategy.

---

### 8. No output validation before writing `research.md`

The agent writes `research.md` following a template, but there's no self-check that the output actually conforms to the template. No validation that all required sections are present. No check that the frontmatter fields are correctly populated.

**Failure scenario:** The agent encounters an unusual codebase, gets confused mid-execution, and writes a `research.md` that's missing the Quality Commands table entirely. The `<mandatory>` tag says "NEVER omit" but there's no enforcement mechanism — it's just a strongly-worded instruction. The product-manager agent downstream parses the file, finds no quality commands section, and either crashes or silently skips verification setup.

**Fix:** Add an explicit self-validation step after writing: re-read the file, verify all required sections exist, verify the frontmatter is parseable. If validation fails, fix or report — don't just hope the LLM followed instructions.

---

### 9. Human approval gate is trivially bypassable and has no timeout

The spec sets `awaitingApproval = true` in a JSON file. But there's no mechanism described for:
- How the human is notified that approval is needed
- What happens if the human never approves (the pipeline hangs forever)
- What prevents another agent from just setting it back to `false`
- What the human is supposed to review (the full `research.md`? a summary?)

**Failure scenario:** Research completes at 2 AM. No notification is sent. The pipeline sits idle for 12 hours. Or: an overeager orchestrator sees the phase is "research" and the file exists, ignores the approval flag, and proceeds with unreviewed research.

**Fix:** Specify the notification mechanism. Define a timeout or escalation path. Specify what "approval" concretely means (which file to check, which field to look for, what values indicate approve vs. reject). Add a review checklist for the human.

---

### 10. Web search queries are too generic to be useful

```
WebSearch: "[topic] best practices"
WebSearch: "[topic] common pitfalls"
```

**Failure scenario:** For a topic like "authentication," these queries return thousands of generic blog posts about password hashing. The agent dutifully records 5 generic findings that add zero value. Meanwhile, the actual relevant concern — like "OAuth 2.1 deprecates implicit flow" — is never found because the query wasn't specific enough.

**Fix:** Require the agent to formulate queries based on the specific technology stack and constraints from `idea.md`, not just the topic. Example: instead of "authentication best practices," search for "NextJS App Router authentication middleware patterns 2025." Also specify how many searches to run and how to evaluate relevance of results before recording them.

---

### 11. The `<mandatory>` tags are a hope-based enforcement mechanism

Four sections are wrapped in `<mandatory>` tags. In an LLM agent, XML tags are stylistic — they carry no execution guarantee. The agent might skip any of these sections and the system has no way to detect it.

**Fix:** If these are truly mandatory, enforce them structurally: have the orchestrator verify the output file contains the required sections before advancing the pipeline state. Don't rely on the agent policing itself through prompt formatting.

---

### Summary of top 3 changes that would most improve this spec:

1. **Add error handling for every tool call** — define what failure looks like and what the agent does about it. This is the single biggest gap.
2. **Define the `idea.md` schema** — garbage in, garbage out. The entire pipeline's accuracy depends on correct interpretation of this file.
3. **Add output validation** — the agent should verify its own output before writing state. The orchestrator should verify it again before advancing. Hope is not a strategy.

## Round 1 — DEFENDER

# Defense: `researcher` Agent Prompt — Round 1

## The Concern the Critic Missed

Before addressing the 11 points: the critic spent considerable energy on edge cases and failure modes but **completely missed the most operationally significant gap** — **there is no guidance on research depth calibration relative to project scope**.

A trivial feature ("add a dark mode toggle") and a complex system ("build a multi-tenant auth layer") both receive the identical 5-step research process. This means either:
- Small projects get over-researched, burning tokens and time on unnecessary web searches and exhaustive codebase scans (directly violating the project's own principle #10: *"cada token cuesta plata"*), or
- Complex projects get the same shallow pass as trivial ones and downstream agents make decisions on insufficient evidence.

This affects **every single execution**, not just edge cases. The critic's top 3 suggested changes are all about failure handling; none address the quality of the *happy path*, which is where this agent spends 95%+ of its runs.

**If I were to propose one change** to this prompt, it would be: after extracting Vision and Constraints from `idea.md`, have the agent classify complexity (trivial/moderate/complex) and scale research depth accordingly — number of searches, depth of codebase exploration, granularity of feasibility assessment.

## Point-by-Point Response

The hardcoded `/tmp/sd-state.json` path is a legitimate bug. I concede this specific issue and the fix is straightforward: use `mktemp`.

However, the **concurrency scenario is dramatically overstated**. This agent operates within a sequential spec-driven pipeline where phases execute one at a time — the whole point of the `awaitingApproval` gate is to prevent the next phase from starting. Two researchers don't run on the same spec simultaneously; the orchestrator prevents it. The critic invented a scenario that the system's architecture already prohibits.

The "state file doesn't exist" concern is more reasonable, but also edge-case territory — the orchestrator creates the state file when initializing a spec. If it's missing, something upstream already failed catastrophically.

**Accepted fix:** Replace the hardcoded temp path with `mktemp`:
```bash

```

That's it. No locking framework needed. The concurrency concern is an architecture misunderstanding.

### 2. No error handling for tool calls — REJECT (mostly)

The critic treats this LLM agent prompt as if it were a bash script or a deterministic program. It is not. **This is a prompt for Claude** — an LLM that inherently handles ambiguity, empty results, and unexpected outputs through judgment.

Specific rebuttals:
- **"WebSearch returns nothing"**: Claude doesn't silently produce an empty section. It writes "No relevant results found for [query]" and moves on. That's what LLMs do with empty results — they narrate them. The constraint "NEVER fabricate sources" + "say 'not found' explicitly" already covers this.
- **"Glob returns 10,000 files"**: The tools themselves have pagination and limits built in. The `Glob` tool returns files sorted by modification time. Claude naturally samples and prioritizes rather than trying to process 10,000 files.
- **"Read on a binary file"**: The Read tool handles this — it returns a notice that the file is binary. Claude doesn't attempt to analyze garbage.
- **"WebFetch on a login wall"**: Claude can recognize when HTML content is a login page rather than documentation. This is exactly the kind of judgment that makes LLM agents useful.

The critic is applying **deterministic systems thinking** to a **probabilistic agent**. Adding explicit error handling for every tool call would triple the prompt length, consume precious context, and make the prompt harder to follow — all for scenarios the agent already handles through basic reasoning.

**One concession**: adding a single line to the Constraints section — *"If a tool call returns empty or irrelevant results, note this explicitly in the output rather than omitting the section"* — would be a proportional improvement. A full error-handling specification for every call would be over-engineering.

## Part A: The Conceded Fixes Are Inadequate

### 1. The `mktemp` fix doesn't address the actual failure mode

The defender accepted replacing the hardcoded `/tmp/sd-state.json` with `mktemp`. Good. But then dismissed the "state file doesn't exist" scenario as "edge-case territory — the orchestrator creates the state file when initializing a spec."

**The prompt doesn't say that.** The prompt tells the researcher agent to run `jq` against a file whose existence is *assumed but never verified*. The defender is defending the prompt by citing orchestrator behavior that exists nowhere in this document. If the orchestrator always creates the file, great — but the researcher prompt should still guard against its absence, because:

1. The prompt is the contract for this agent's behavior. It shouldn't depend on undocumented guarantees from other components.
2. `jq` reading a nonexistent file produces an error, the `&&` chain halts, `awaitingApproval` never gets set, and the pipeline hangs silently — the exact failure mode the defender claims the approval gate prevents.

**The fix that actually works:**
```bash
state_file="{basePath}/.spec-drive-state.json"
[ -f "$state_file" ] || echo '{}' > "$state_file"
tmpfile=$(mktemp) && jq '.awaitingApproval = true' "$state_file" > "$tmpfile" && mv "$tmpfile" "$state_file"
```

Three lines instead of one. Not "a locking framework" — just basic defensive scripting.

### 4. "Add multi-language discovery guidance" is a one-liner that solves nothing

The defender's accepted fix: *"Add a line: 'If the project is not Node.js-based, look for equivalent tooling: `pyproject.toml`/`setup.cfg` (Python), `Makefile`/`Cargo.toml` (Rust), `go.mod` (Go).'"*

Adding a parenthetical "oh and also check these other files" doesn't change the execution flow. The agent still starts with package.json, finds nothing, and then what? The prompt doesn't say "detect the project language first, then choose your discovery path." It says "check these sources in order" and then adds a footnote.

**What would actually work:** Restructure Step 3 to start with language detection:

> First, determine the project's primary language/runtime by examining file extensions, lock files (`package-lock.json`, `poetry.lock`, `Cargo.lock`, `go.sum`), and configuration files. Then select the appropriate quality command sources for that ecosystem.

This changes the *control flow*, not just the reference material.

### 6. Calibration anchors are better but still ungrounded

The defender accepted adding:
```
S = single file/function | M = multiple files, no new deps | L = new module or integration | XL = new infrastructure
```

**New problem introduced:** These anchors describe *scope*, not *effort*. A single-file change to a payment processing function with complex business logic is not "S = hours." A new module that's a thin wrapper around a well-documented API might be "M" in scope but "S" in effort. The defender conflated the two and the anchors will mislead.

More critically: **the agent has no basis for estimating effort because it hasn't seen the team's velocity.** Is this a solo developer or a 10-person team? Are they familiar with the codebase or onboarding? The researcher has access to the codebase and the web — it does not have access to team composition, skill level, or availability. Any effort estimate it produces is a guess dressed up as analysis.

**Better approach:** Drop effort estimation entirely from the researcher. The researcher should output **scope indicators** (files affected, new dependencies required, external integrations needed) and let the task-planner — which has a better vantage point — derive effort. The researcher produces *evidence*; the planner makes *estimates*. That's the separation of concerns the defender keeps invoking.

### 8. Self-check is the right idea, implemented wrong

The defender accepted: *"Before setting awaitingApproval, re-read research.md and verify: frontmatter is present, all five sections exist, and no section is empty."*

**This is asking the same model that just wrote the document to audit the document it just wrote.** The defender even acknowledged this: *"asking an LLM to validate its own output is only marginally more reliable than asking it to produce correct output in the first place."*

So the defender conceded the point, proposed a fix they themselves called marginally reliable, and moved on. That's not a fix — it's a checkbox.

**What would actually add value:** Instead of a generic "verify sections exist" instruction, specify a *structured output* as the final action — a JSON manifest that the orchestrator can mechanically validate:

```json
{
  "sectionsPresent": ["executiveSummary", "externalResearch", "codebaseAnalysis", "qualityCommands", "feasibilityAssessment", "openQuestions"],
  "qualityCommandsFound": 3,
  "openQuestionsCount": 2,
  "webSearchesPerformed": 4,
  "filesAnalyzed": 12
}
```

Write this to `{basePath}/.research-manifest.json`. Now the orchestrator has something concrete to validate — not "did the LLM say it checked itself?" but "does this manifest indicate complete research?" This is the structural enforcement the defender keeps saying belongs in the orchestrator — fine, but then **give the orchestrator something to enforce against**.

## Part B: The "Claude is Smart Enough" Consistency Trap

This is the critic's strongest new argument and deserves a serious response.

> The defender is simultaneously arguing that this prompt is well-crafted AND that half its specificity is unnecessary because the LLM will do the right thing anyway.

This is a mischaracterization, but it reveals a real tension I should articulate clearly.

The prompt specifies two different categories of things:

1. **Structure**: What steps to perform, what output format to produce, what sections to include, what constraints to respect. These define the *shape* of the agent's work. Without them, the agent produces unstructured, inconsistent output that downstream agents can't reliably parse. **Structure must be specified.**

2. **Judgment within structure**: How to handle an empty search result, which file extensions to glob for in an unfamiliar codebase, how to formulate a specific search query from a general topic. These are decisions *within* a defined step. **Judgment should be left to the LLM.**

The distinction isn't "specificity matters vs. specificity doesn't matter." It's: **specify the what, leave the how.** The prompt correctly specifies *what* to search for (best practices, pitfalls, official docs) and leaves *how* to construct the query to the agent. It correctly specifies *what* to look for in the codebase (similar code, naming conventions, test patterns) and leaves *how* to navigate an unfamiliar codebase to the agent.

Where I conceded in Round 1 and Round 2, it's because the critic identified **structural gaps** — missing steps, wrong control flow (language detection before quality commands), missing output sections (interpretation echo). Where I rejected, it's because the critic wanted to specify **judgment** — what to do when a search returns empty, how to handle binary files, how to construct queries.

The critic asks: *"If Claude is smart enough, why have a prompt at all?"* The answer: to define structure and constraints. Not to script every conditional branch. A recipe says "sauté until golden" — it doesn't say "if the garlic burns, reduce heat; if the oil pops, step back; if the pan warps, use a different pan." The chef brings judgment. The recipe brings structure.

This is why I keep rejecting the "add error handling for every tool call" argument. It would convert the prompt from a **structural specification** into a **behavioral script**, making it longer, more brittle, and harder to maintain — all for marginal reliability gains in scenarios the LLM already handles through judgment.
